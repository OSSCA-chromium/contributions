<p>이 문서는 <strong>Running and Debugging Web Tests</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_tests.md"><code>docs/testing/web_tests.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p>웹 테스트는 Blink에서 레이아웃과 렌더링을 포함하되 이에 국한되지 않는 여러 컴포넌트를 테스트하는 데 사용된다. 일반적으로 웹 테스트는 테스트 렌더러(<code>content_shell</code>)에서 페이지를 로드하고 렌더링된 출력 또는 JavaScript 출력을 기대 출력 파일과 비교하는 방식으로 이루어진다.</p>
<p>이 문서는 기존 웹 테스트를 실행하고 디버깅하는 방법을 다룬다. 웹 테스트를 작성해야 하는 상황이라면 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/writing_web_tests.md">Writing Web Tests 문서</a>를 참고하라.</p>
<p>"layout tests"라는 용어를 "web tests"로 변경했다는 점에 유의하라. 이 용어들은 같은 것을 의미한다고 보면 된다. 또한 이를 "WebKit tests" 및 "WebKit layout tests"라고도 부른다.</p>
<p><a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_platform_tests.md">"Web platform tests"</a>(WPT)는 권장되는 웹 테스트 형식이며 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/external/wpt">web_tests/external/wpt</a>에 위치한다. 브라우저 간에 동작해야 하는 테스트는 그곳에 둔다. 다른 디렉터리는 Chrome 전용 테스트만을 위한 것이다.</p>
<p>참고: Web Platform Test에 대한 가이드를 찾고 있다면 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_platform_tests.md">"Web platform tests"</a>(WPT)를 읽어야 한다. 이 문서는 WPT 고유 기능/동작을 다루지 않는다.</p>
<p>참고: Chrome, Chrome Android 또는 WebView로 Web Platform Tests를 실행하는 가이드를 찾고 있다면 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/run_web_platform_tests.md">"Running Web Platform Tests with run_wpt_tests.py"</a>를 읽어야 한다.</p>
<h2 id="웹-테스트-실행하기">웹 테스트 실행하기</h2><h3 id="지원-플랫폼">지원 플랫폼</h3><ul>
<li>Linux</li>
<li>MacOS</li>
<li>Windows</li>
<li>Fuchsia</li>
</ul>
<p>Android는 <a href="https://crbug.com/567947">지원되지 않는다</a>.</p>
<h3 id="초기-설정">초기 설정</h3><p>웹 테스트를 실행하려면 먼저 <code>blink_tests</code> 타깃을 빌드하여 <code>content_shell</code> 및 필요한 다른 모든 바이너리를 얻어야 한다.</p>
<pre><code class="hljs language-bash">autoninja -C out/Default blink_tests</code></pre><p><strong>Mac</strong>에서는 테스트를 시작하기 전에 content_shell 바이너리에서 심볼을 제거(strip)하는 것이 좋다. 그렇게 하지 않으면 5~10개의 프로세스가 동시에 실행되면서 모두 OS 크래시 리포터의 검사를 받느라 멈춰 있을 수 있다. 이로 인해 일반적으로는 발생하지 않는 타임아웃 같은 다른 실패가 발생할 수 있다.</p>
<pre><code class="hljs language-bash">strip ./out/Default/Content\ Shell.app/Contents/MacOS/Content\ Shell</code></pre><h3 id="테스트-실행하기">테스트 실행하기</h3><p>테스트 러너 스크립트는 <code>third_party/blink/tools/run_web_tests.py</code>에 있다.</p>
<p>사용할 빌드 디렉터리(예: out/Default 등)를 지정하려면 <code>-t</code> 또는 <code>--target</code> 매개변수를 전달해야 한다. 디렉터리를 지정하지 않으면 <code>out/Release</code>가 사용된다. 내장된 <code>out/Default</code>를 사용하려면 다음을 사용한다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py -t Default</code></pre><p>*** promo</p>
<ul>
<li>Windows 사용자는 대신 <code>third_party\blink\tools\run_web_tests.bat</code>를 사용해야 한다.</li>
<li>Linux 사용자는 <code>testing/xvfb.py</code>를 사용해서는 안 된다. <code>run_web_tests.py</code>가 Xvfb 자체를 관리한다.</li>
</ul>
<hr />
<p><a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/TestExpectations">TestExpectations</a>에서 <code>[ Skip ]</code>으로 표시된 테스트는 기본적으로 실행되지 않는다. 일반적으로 다루기 어려운 도구 오류를 일으키기 때문이다. 그중 하나를 강제로 실행하려면 해당 파일 이름을 바꾸거나, 건너뛴 테스트를 명령줄(아래 참조)에 지정하거나, --test-list로 지정한 파일에 넣으면 된다(단, --skip=always는 <code>[ Skip ]</code>으로 표시된 테스트를 항상 건너뛰게 만들 수 있다). TestExpectations 및 관련 파일에 대해 자세히 알아보려면 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_test_expectations.md">Web Test Expectations 문서</a>를 읽어라.</p>
<p>*** promo<br />현재 Fuchsia 봇에서는 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/TestLists/Default.txt">Default.txt</a>에 나열된 테스트만 실행된다. 모든 웹 테스트를 실행하는 데 Fuchshia에서 시간이 너무 오래 걸리기 때문이다. 대부분의 개발자는 Blink 테스트를 Linux에 집중한다. 우리는 스모크 테스트가 다루는 시나리오를 벗어난 경우 Linux와 Fuchsia의 동작이 거의 동일하다는 사실에 의존한다.</p>
<hr />
<p>*** promo<br />Fuchsia의 경우와 유사하게, 오래된 mac 버전 봇에서는 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/TestLists/MacOld.txt">MacOld.txt</a>에 나열된 테스트가 실행된다. 이렇게 함으로써 테스트 실행에 필요한 리소스를 줄였다. 이는 대부분의 웹 테스트가 서로 다른 mac 버전에서도 유사하게 동작한다는 사실에 의존한다.</p>
<hr />
<p>일부 테스트만 실행하려면 웹 테스트 디렉터리(<code>src/third_party/blink/web_tests</code>)를 기준으로 한 디렉터리나 파일 이름을 <code>run_web_tests.py</code>의 인수로 지정한다. 예를 들어 fast form 테스트를 실행하려면 다음을 사용한다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py fast/forms</code></pre><p>또는 다음 단축형을 사용할 수도 있다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py fast/fo\*</code></pre><p>*** promo<br />예: <code>content_shell</code>의 디버그 빌드로 웹 테스트를 실행하되 SVG 테스트만 테스트하고 픽셀 테스트를 실행하려면 다음을 실행한다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py -t Default svg</code></pre><hr />
<p>마지막으로 빠르지만 덜 견고한 대안으로, content_shell 실행 파일을 직접 사용하여 특정 테스트를 실행할 수도 있다(Windows 예시).</p>
<pre><code class="hljs language-bash">out\Default\content_shell.exe --run-web-tests &lt;url&gt;|&lt;full_test_source_path&gt;|&lt;relative_test_path&gt;</code></pre><p>예:</p>
<pre><code class="hljs language-bash">out\Default\content_shell.exe --run-web-tests \
    c:\chrome\src\third_party\blink\web_tests\fast\forms\001.html</code></pre><p>또는</p>
<pre><code class="hljs language-bash">out\Default\content_shell.exe --run-web-tests fast\forms\001.html</code></pre><p>하지만 이 방법은 셸이 대신 해주지 않기 때문에 기대 결과와 수동으로 diff를 비교해야 한다. 또한 텍스트 결과만 덤프한다(픽셀 및 오디오 바이너리 데이터의 덤프는 사람이 읽을 수 없기 때문이다). <code>content_shell</code> 실행에 대한 자세한 내용은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_tests_in_content_shell.md">Running Web Tests Using the Content Shell</a>을 참고하라.</p>
<p>지원되는 인수의 전체 목록을 보려면 다음을 실행한다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py --<span class="hljs-built_in">help</span></code></pre><p>*** note<br /><strong>Linux 참고:</strong> 우리는 글꼴 메트릭과 위젯 메트릭을 맞춰 Windows 렌더 트리 출력과 정확히 일치시키려고 한다. 렌더 트리 출력에 차이가 있다면, 글꼴 메트릭을 개선하여 리베이스라인을 피할 수 있는지 확인해야 한다. Linux 웹 테스트에 대한 추가 정보는 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_tests_linux.md">docs/web_tests_linux.md</a>를 참고하라.</p>
<hr />
<p>*** note<br /><strong>Mac 참고:</strong> 테스트가 실행되는 동안 올바른 종류의 스크롤바, 색상 등이 사용되도록 여러 Appearance 설정이 자동으로 덮어써진다. 픽셀 테스트에서 기대하는 것과 ColorSync의 색상 보정이 일치하도록 기본 디스플레이의 "Color Profile"도 변경된다. 이 변경은 눈에 띄며, 그 정도는 디스플레이의 일반적인 보정 수준에 따라 달라진다. 테스트는 완료 시 설정을 복원하기 위해 최선을 다하지만, 잘못된 상태로 남아 있다면 System Preferences → Displays → Color로 이동해 "올바른" 값을 선택하여 수동으로 재설정할 수 있다.</p>

<h2 id="성공과-실패">성공과 실패</h2><p>테스트의 출력이 미리 정의된 기대 결과와 일치하면 테스트는 성공한다. 테스트가 하나라도 실패하면 테스트 스크립트는 실제로 생성된 결과와 실제 결과/기대 결과의 diff를 <code>src/out/Default/layout-test-results/</code>에 넣고, 기본적으로 결과/diff에 대한 요약과 링크가 있는 브라우저를 실행한다.</p>
<p>테스트의 기대 결과는 <code>src/third_party/blink/web_tests/platform</code> 또는 각 테스트 옆에 있다.</p>
<p>*** note<br /><a href="https://github.com/w3c/testharness.js/">testharness.js</a>를 사용하는 테스트는 모든 테스트 케이스가 통과하면 기대 결과 파일을 갖지 않는다.</p>
<hr />
<p>실행되지만 잘못된 출력을 생성하는 테스트는 "failed"로 표시되고, 테스트 셸을 크래시시키는 테스트는 "crashed"로 표시되며, 완료하는 데 일정 시간보다 오래 걸리는 테스트는 중단되어 "timed out"으로 표시된다. 스크립트 출력의 점 행은 하나 이상의 테스트가 통과했음을 나타낸다.</p>
<h2 id="테스트-기대값">테스트 기대값</h2><p><a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/TestExpectations">TestExpectations</a> 파일(및 관련 파일)은 알려진 모든 웹 테스트 실패 목록을 포함한다. 이에 대한 자세한 내용은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_test_expectations.md">Web Test Expectations 문서</a>를 참고하라.</p>
<h2 id="테스트-런타임-플래그">테스트 런타임 플래그</h2><p>추가 명령줄 인수로 웹 테스트를 실행하는 방법은 두 가지가 있다.</p>
<h3 id="--flag-specific">--flag-specific</h3><pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py --flag-specific=blocking-repaint</code></pre><p>이를 위해서는 <code>web_tests/FlagSpecificConfig</code>에 다음과 같은 항목이 있어야 한다.</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  <span class="hljs-attr">"name"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"blocking-repaint"</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"args"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"--blocking-repaint"</span><span class="hljs-punctuation">,</span> <span class="hljs-string">"--another-flag"</span><span class="hljs-punctuation">]</span>
<span class="hljs-punctuation">}</span></code></pre><p>이는 테스트 하니스가 <code>--blocking-repaint --another-flag</code>를 content_shell 바이너리에 전달하도록 지시한다.</p>
<p>또한 이 파일이 존재한다면 <code>web_tests/FlagExpectations/blocking-repaint</code>에서 플래그별 기대값을 찾는다. 이 파일의 suppression은 기본 TestExpectations 파일을 재정의한다. 그러나 플래그별 기대값 또는 기본 기대값 중 어느 쪽에 있는 <code>[ Slow ]</code>도 항상 사용되는 기대값에 병합된다.</p>
<p>또한 <code>web_tests/flag-specific/blocking-repaint</code>에서 베이스라인을 찾는다. 이 디렉터리의 베이스라인은 폴백 베이스라인을 재정의한다.</p>
<p>*** note<br /><a href="https://chromium.googlesource.com/chromium/src/+/main/BUILD.gn">BUILD.gn</a>은 플래그별 빌더가 항상 linux 봇에서 실행된다고 가정하므로, 플래그별 테스트 기대값과 베이스라인은 linux 봇에만 다운로드된다. 다른 플랫폼에서 플래그별 빌더를 실행해야 한다면, 해당 플랫폼으로 플래그별 관련 데이터를 다운로드하도록 BUILD.gn을 업데이트하라.</p>
<hr />
<p><code>--additional-driver-flag</code>를 사용해 content_shell에 추가 명령줄 인수를 지정할 수도 있지만, 이 경우 테스트 하니스는 플래그별 테스트 기대값이나 베이스라인을 사용하지 않는다.</p>
<h3 id="가상-테스트-스위트">가상 테스트 스위트</h3><p><em>가상 테스트 스위트</em>는 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/VirtualTestSuites">web_tests/VirtualTestSuites</a>에 정의할 수 있으며, 추가 플래그와 함께 웹 테스트의 하위 집합을 실행하고, 경로에는 <code>virtual/&lt;prefix&gt;/...</code>를 사용한다. 테스트는 지정된 bases 중 하나와 경로가 일치하는 실제 기본 테스트(디렉터리 또는 파일)에 매핑되는 가상 테스트일 수도 있고, <code>web_tests/virtual/&lt;prefix&gt;/</code> 디렉터리 아래의 실제 테스트일 수도 있다. 예를 들어, 다음 가상 테스트 스위트를 사용해 repainting을 위한 (가상의) 새 모드를 테스트할 수 있다.</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  <span class="hljs-attr">"prefix"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"blocking_repaint"</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"platforms"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"Linux"</span><span class="hljs-punctuation">,</span> <span class="hljs-string">"Mac"</span><span class="hljs-punctuation">,</span> <span class="hljs-string">"Win"</span><span class="hljs-punctuation">]</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"bases"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"compositing"</span><span class="hljs-punctuation">,</span> <span class="hljs-string">"fast/repaint"</span><span class="hljs-punctuation">]</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"args"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"--blocking-repaint"</span><span class="hljs-punctuation">]</span>
<span class="hljs-punctuation">}</span></code></pre><p>이는 각각 <code>web_tests/compositing</code> 및 <code>web_tests/fast/repaint</code> 아래 파일에 대응하는 <code>virtual/blocking_repaint/compositing/...</code> 및 <code>virtual/blocking_repaint/fast/repaint/...</code> 형식의 새 "가상" 테스트를 만들고, 실행될 때 <code>content_shell</code>에 <code>--blocking-repaint</code>를 전달한다.</p>
<p>다음 명령줄로 테스트를 실행할 수 있다는 점에 유의하라.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_web_tests.py virtual/blocking_repaint/compositing \
  virtual/blocking_repaint/fast/repaint</code></pre><p>이러한 가상 테스트는 원래의 <code>compositing/...</code> 및 <code>fast/repaint/...</code> 테스트에 추가로 존재한다. 이들은 <code>web_tests/TestExpectations</code>에 자체 기대값을 가질 수 있고, 자체 베이스라인도 가질 수 있다. 테스트 하니스는 비가상 기대값과 베이스라인을 폴백으로 사용한다. 가상 테스트에 자체 기대값이 있으면 모든 비가상 기대값을 재정의한다. 그렇지 않으면 비가상 기대값이 사용된다. 그러나 가상 또는 비가상 기대값 중 어느 쪽에 있는 <code>[ Slow ]</code>도 항상 사용되는 기대값에 병합된다. 가상 테스트는 통과할 것으로 기대되지만 비가상 테스트는 실패할 것으로 기대되는 경우, 가상 테스트에 대해 명시적인 <code>[ Pass ]</code> 항목을 추가해야 한다.</p>
<p>이렇게 하면 <code>web_tests/virtual/blocking_repaint</code> 디렉터리 아래의 실제 테스트도 <code>--blocking-repaint</code> 플래그로 실행될 수 있다.</p>
<p>"platforms" 설정은 일부 플랫폼에서 테스트를 건너뛰는 데 사용할 수 있다. 가상 테스트 스위트가 전체 테스트 시간의 5% 이상을 사용한다면 일부 플랫폼에서 해당 테스트 스위트를 건너뛰는 것을 고려해야 한다.</p>
<p>"prefix" 값은 고유해야 한다. 같은 플래그를 사용하는 여러 디렉터리는 같은 "bases" 목록에 나열해야 한다. 단지 <code>virtual/&lt;prefix&gt;</code> 아래의 실제 테스트를 플래그와 함께 실행하고 가상 테스트는 만들고 싶지 않은 경우, "bases" 목록은 비어 있을 수 있다.</p>
<p>가상 테스트 스위트는 선택적 <code>exclusive_tests</code> 필드를 가질 수 있으며, 이 필드는 이 가상 스위트 아래에서만 독점적으로 실행될 모든(<code>"ALL"</code>) 또는 일부 <code>bases</code> 테스트를 지정한다. 지정된 기본 테스트는 건너뛴다. 해당 테스트를 <code>exclusive_tests</code> 목록에 지정하지 않은 다른 가상 스위트 아래의 대응 가상 테스트도 건너뛴다. 예를 들어(관련 없는 필드는 생략):</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  <span class="hljs-attr">"prefix"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"v1"</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"bases"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"a"</span><span class="hljs-punctuation">]</span><span class="hljs-punctuation">,</span>

  <span class="hljs-attr">"bases"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"a/a1"</span><span class="hljs-punctuation">,</span> <span class="hljs-string">"a/a2"</span><span class="hljs-punctuation">]</span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">"exclusive_tests"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"ALL"</span><span class="hljs-punctuation">,</span>
<span class="hljs-punctuation">}</span>

<h3 id="flag-specific과-가상-테스트-스위트-중-선택하기">flag-specific과 가상 테스트 스위트 중 선택하기</h3><p>구현이 아직 진행 중인 플래그의 경우, 플래그별 기대값과 가상 테스트 스위트는 활성화된 코드 경로와 활성화되지 않은 코드 경로를 모두 테스트하기 위한 두 가지 대안 전략을 나타낸다. 기능이 프로덕션과 실질적으로 다른 코드 경로를 갖는다면 단순히 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/RuntimeEnabledFeatures.md">runtime enabled feature</a>를 <code>status: "test"</code>로 설정하는 것보다 이 방법들이 선호된다. 후자는 프로덕션 코드 경로의 테스트 커버리지를 잃게 만들기 때문이다.</p>
<p>가상 테스트 스위트와 플래그별 스위트 중 선택할 때 다음을 고려하라.</p>
<ul>
<li><p><a href="https://dev.chromium.org/developers/testing/chromium-build-infrastructure/tour-of-the-chromium-buildbot">waterfall builders</a>와 <a href="https://dev.chromium.org/developers/testing/try-server-usage">try bots</a>는 비가상 테스트에 더해 모든 가상 테스트 스위트를 실행한다. 반대로 플래그별 설정은 봇이 자동으로 해당 플래그를 테스트하게 만들지 않는다. 가상 테스트 스위트 없이 봇 커버리지를 원한다면 <a href="#cqci%EC%97%90%EC%84%9C-%EC%83%88-flag-specific-%EC%8A%A4%EC%9C%84%ED%8A%B8-%EC%8B%A4%ED%96%89%ED%95%98%EA%B8%B0">이 지침</a>을 따라야 한다.</p>
</li>
<li><p>위의 이유로, 가상 테스트 스위트는 커밋 큐와 지속적 빌드 인프라에 성능 비용을 발생시킨다. 플래그가 변경될 때마다 <code>content_shell</code>을 재시작해야 하므로 병렬성이 제한되어 이 비용은 더 커진다. 따라서 많은 수의 가상 테스트 스위트를 추가하는 것은 피해야 한다. 가상 테스트 스위트는 기능과 직접 관련된 테스트 하위 집합을 실행하는 데 적합하지만, 모든 테스트에 잠재적으로 영향을 주는 깊은 아키텍처 변경을 만드는 플래그에는 확장성이 없다.</p>
</li>
<li><p>가상 테스트 경로 이름에서 와일드카드(예: <code>virtual/blocking_repaint/fast/repaint/*</code>)를 사용하는 것은 <code>run_web_tests.py</code> 명령줄에서 지원되지 않는다는 점에 유의하라. 그러나 여전히 <code>virtual/blocking_repaint</code>를 사용해 스위트의 모든 실제 및 가상 테스트를 실행하거나, <code>virtual/blocking_repaint/fast/repaint/dir</code>을 사용해 특정 디렉터리 아래 스위트의 실제 또는 가상 테스트를 실행할 수 있다.</p>
</li>
</ul>
<p>*** note<br />추가 플래그와 함께 가상 테스트를 실행할 수 있다. 가상 args와 추가 플래그가 모두 적용된다. 베이스라인과 기대값의 폴백 순서는 다음과 같다. 1) 플래그별 가상, 2) 비플래그별 가상, 3) 플래그별 기본, 4) 비플래그별 기본</p>
<hr />
<h3 id="cqci에서-새-flag-specific-스위트-실행하기">CQ/CI에서 새 Flag-Specific 스위트 실행하기</h3><p>이미 <code>FlagSpecificConfig</code> 항목을 만들었다고 가정한다.</p>
<ol>
<li><code>chromium.tests</code> swarming 풀의 용량 증가를 위해 리소스 요청(<a href="https://g3doc.corp.google.com/company/teams/chrome/ops/business/resources/resource-request-program.md?cl=head&amp;polyglot=chrome-browser#i-need-new-resources">내부 문서</a>)을 제출하고 승인을 기다린다.</li>
<li><code>--flag-specific</code> 및 필요하다면 다른 특수 설정(예: 더 적은 shard)을 가진 새 전용 <a href="https://source.chromium.org/chromium/chromium/src/+/main:testing/buildbot/test_suites.pyl;l=1516-1583;drc=0694b605fb77c975a065a3734bdcf3bd81fd8ca4;bpv=0;bpt=0">Buildbot 테스트 스위트</a>를 정의한다.</li>
<li>먼저 Buildbot 스위트를 관련 <code>*-blink-rel</code> 빌더의 composition suite에 추가한다(<a href="https://source.chromium.org/chromium/chromium/src/+/main:testing/buildbot/test_suites.pyl;l=5779-5780;drc=0694b605fb77c975a065a3734bdcf3bd81fd8ca4;bpv=0;bpt=0">예시</a>).</li>
<li>관련 빌더의 <a href="https://source.chromium.org/chromium/chromium/src/+/main:third_party/blink/tools/blinkpy/common/config/builders.json;l=127-129;drc=ff938aaff9566b2cc442476a51835e0b90b1c6f6;bpv=0;bpt=0"><code>builders.json</code></a>에 플래그별 step 이름을 추가한다. 이제 <code>rebaseline-cl</code>과 WPT importer가 해당 스위트의 베이스라인을 생성한다.</li>
<li>새 스위트를 리베이스라인하고 <code>FlagExpectations/</code> 아래에 필요한 suppression을 추가한다.</li>
<li>원하는 빌더에 Buildbot 스위트를 추가하여 CQ/CI에서 플래그별 스위트를 활성화한다. 이는 <a href="https://source.chromium.org/chromium/chromium/src/+/main:testing/buildbot/test_suites.pyl;l=5828-5829;drc=0694b605fb77c975a065a3734bdcf3bd81fd8ca4;bpv=0;bpt=0"><code>linux-rel</code></a> 같은 기존 CQ 빌더일 수도 있고, 맞춤 위치 필터가 있는 <a href="https://source.chromium.org/chromium/chromium/src/+/main:infra/config/subprojects/chromium/try/tryserver.chromium.accessibility.star;drc=adad4c6d55e69783ba1f16d30f4bc7367e2e626a;bpv=0;bpt=0"><code>linux-blink-web-tests-force-accessibility-rel</code></a> 같은 전용 빌더일 수도 있다.</li>
</ol>
<h2 id="테스트-실패-추적하기">테스트 실패 추적하기</h2><p>웹 테스트 실패와 관련된 모든 버그에는 <a href="https://crbug.com/?q=label:Test-Layout">Test-Layout</a> 라벨이 있어야 한다. 버그에 대해 얼마나 알고 있는지에 따라 상태를 적절히 지정한다.</p>
<ul>
<li><strong>Unconfirmed</strong> -- 이것이 단순 리베이스라인인지, 기존 버그의 가능한 중복인지, 실제 실패인지 확실하지 않다.</li>
<li><strong>Untriaged</strong> -- 확인되었지만 우선순위나 근본 원인이 불확실하다.</li>
<li><strong>Available</strong> -- 문제의 근본 원인을 알고 있다.</li>
<li><strong>Assigned</strong> 또는 <strong>Started</strong> -- 이 문제를 수정할 예정이다.</li>
</ul>
<p>새 웹 테스트 버그를 만들 때는 다음 속성을 설정하라.</p>
<ul>
<li>Components: Blink의 하위 컴포넌트</li>
<li>OS: <strong>All</strong>(또는 실패가 발생한 OS)</li>
<li>Priority: 2(크래시라면 1)</li>
<li>Type: <strong>Bug</strong></li>
<li>Labels: <strong>Test-Layout</strong></li>
</ul>
<p>이 라벨들을 미리 설정해 주는 <em>Layout Test Failure</em> 템플릿을 사용할 수도 있다.</p>
<h2 id="웹-테스트-디버깅하기">웹 테스트 디버깅하기</h2><p>웹 테스트가 실행된 후에는 통과하거나 실패한 테스트의 요약을 받게 된다. 무언가가 예기치 않게 실패하면(새 회귀), 예상치 못한 실패의 요약이 있는 <code>content_shell</code> 창이 나타난다. 또는 조사하려는 실패 테스트가 이미 있을 수도 있다. 어떤 경우든 문제를 찾기 위한 몇 가지 단계와 팁은 다음과 같다.</p>
<ul>
<li>결과를 살펴본다. 때로는 테스트가 패치에서 도입된 변경 사항을 반영하기 위해 리베이스라인(아래 참조)만 필요로 한다.<ul>
<li>테스트를 trunk Chrome 또는 content_shell 빌드에 로드하고 결과를 살펴본다. (http/ 디렉터리의 테스트는 먼저 http 서버를 시작한다. 위를 참고하라. <code>http://localhost:8000/</code>로 이동해 거기서부터 진행한다.) 좋은 테스트는 무엇을 확인하는지 설명하지만, 모든 테스트가 그런 것은 아니며, 때로는 명시적으로 테스트하지 않는 것들이 여전히 깨져 있을 수 있다. 필요한 경우 Safari, Firefox, IE와 비교하여 올바른지 확인한다. 그래도 확실하지 않으면 그 내용을 가장 잘 아는 사람을 찾아 물어본다.</li>
<li>일부 테스트는 Chrome이 아니라 content_shell에서만 제대로 동작한다. 그곳에 노출된 추가 API에 의존하기 때문이다.</li>
<li>일부 테스트는 content_shell에 직접 로드했을 때가 아니라 웹 테스트 프레임워크에서 실행될 때만 제대로 동작한다. 테스트의 보이는 텍스트에 그 내용이 언급되어 있어야 하지만, 모든 테스트가 그렇지는 않다. 그러므로 그것도 시도해 보라. 위의 "테스트 실행하기"를 참고하라.</li>
</ul>
</li>
<li>테스트가 올바르다고 생각한다면, 기대 결과와 실제 결과 사이의 diff를 살펴보며 의심을 확인한다.<ul>
<li>보고된 diff가 중요하지 않은지 확인한다. 간격이나 박스 크기의 작은 차이는 특히 글꼴과 폼 컨트롤 주변에서 중요하지 않은 경우가 많다. JS 오류 메시지의 문구 차이도 대개 허용 가능하다.</li>
<li><code>third_party/blink/tools/run_web_tests.py path/to/your/test.html</code>은 모든 테스트 결과를 나열하는 페이지를 생성한다. 기대값에 실패한 테스트에는 기대 결과, 실제 결과, diff에 대한 링크가 포함된다. 이 결과는 <code>$root_build_dir/layout-test-results</code>에 저장된다.<ul>
<li>또는 <code>--results-directory=path/for/output/</code> 옵션을 사용해 출력이 저장될 대체 디렉터리를 지정할 수 있다.</li>
</ul>
</li>
<li>그래도 올바르다고 확신한다면 테스트를 리베이스라인한다(아래 참조). 그렇지 않다면...</li>
</ul>
</li>
<li>운이 좋다면, 테스트는 content_shell에서 일반적으로 탐색했을 때 제대로 실행되는 테스트일 것이다. 이 경우 Debug content_shell 프로젝트를 빌드하고, 선호하는 디버거에서 실행한 뒤, <code>file:</code> URL에서 테스트 파일을 로드한다.<ul>
<li>content_shell을 여러 번 시작하고 중지하게 될 가능성이 높다. VS에서는 매번 테스트로 이동하는 일을 줄이기 위해 content_shell 프로젝트 Properties의 Debugging 섹션에서 명령 인수로 테스트 URL(<code>file:</code> 또는 <code>http:</code>)을 설정할 수 있다.</li>
<li>테스트에 JS 호출, DOM 조작 또는 실패한다고 생각되는 다른 특징적인 코드 조각이 포함되어 있다면 Chrome 솔루션에서 그것을 검색한다. 문제 추적을 시작하기 위한 시작 브레이크포인트를 두기에 좋은 위치다.</li>
<li>그렇지 않다면 Chrome에서와 마찬가지로 표준 메시지 루프에서 실행 중이다. 다른 정보가 없다면 페이지 로드에 브레이크포인트를 설정한다.</li>
</ul>
</li>
<li>테스트가 전체 웹 테스트 모드에서만 동작하거나, 대화형 세션의 모든 오버헤드 없이 디버깅하는 것이 더 간단하다면, <code>--run-web-tests</code> 명령줄 플래그와 테스트에 대한 URL(<code>file:</code> 또는 <code>http:</code>)을 붙여 content_shell을 시작한다. content_shell에서 웹 테스트를 실행하는 방법에 대한 자세한 정보는 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_tests_in_content_shell.md">여기</a>에서 찾을 수 있다.<ul>
<li>VS에서는 content_shell 프로젝트 Properties의 Debugging 섹션에서 이 작업을 할 수 있다.</li>
<li>이제 웹 테스트가 사용하는 것과 정확히 같은 API, 테마 및 기타 설정으로 실행 중이다.</li>
<li>다시 말해, 테스트에 JS 호출, DOM 조작 또는 실패한다고 생각되는 다른 특징적인 코드 조각이 포함되어 있다면 Chrome 솔루션에서 그것을 검색한다. 문제 추적을 시작하기 위한 시작 브레이크포인트를 두기에 좋은 위치다.</li>
<li>브레이크포인트를 설정할 더 나은 위치를 찾을 수 없다면 <code>content_shell_main.cc</code>의 <code>TestShell::RunFileTest()</code> 호출이나 <code>content_shell_win.cc</code>의 <code>shell-&gt;LoadURL() within RunFileTest()</code>에서 시작한다.</li>
</ul>
</li>
<li>평소처럼 디버깅한다. 여기까지 왔다면 실패한 웹 테스트는 문제를 드러내는 (바라건대) 축소된 테스트 케이스일 뿐이다.</li>
</ul>
<h3 id="http-테스트-디버깅하기">HTTP 테스트 디버깅하기</h3><p>참고: HTTP Tests는 WebKit Layout Tests에서 유래한 스위트의 하위 집합인 <code>web_tests/http/tests/</code> 아래의 테스트를 의미한다. WPT의 HTTP 동작을 디버깅하려면 대신 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_platform_tests.md">"Web platform tests"</a>를 읽어야 한다.</p>
<p>실패를 재현/디버깅하기 위해 서버를 수동으로 실행하려면 다음을 사용한다.</p>
<pre><code class="hljs language-bash">third_party/blink/tools/run_blink_httpd.py</code></pre><p>웹 테스트는 <code>http://127.0.0.1:8000/</code>에서 제공된다. 예를 들어 <code>web_tests/http/tests/serviceworker/chromium/service-worker-allowed.html</code> 테스트를 실행하려면 <code>http://127.0.0.1:8000/serviceworker/chromium/service-worker-allowed.html</code>로 이동한다. 일부 테스트는 <code>127.0.0.1</code>로 갈 때와 <code>localhost</code>로 갈 때 다르게 동작하므로 <code>127.0.0.1</code>을 사용하라.</p>
<p>서버를 종료하려면 <code>run_blink_httpd.py</code>가 실행 중인 터미널에서 아무 키나 누르거나, Windows에서는 <code>taskkill</code> 또는 Task Manager를 사용하고, macOS에서는 <code>killall</code> 또는 Activity Monitor를 사용한다.</p>
<p>테스트 서버는 <code>web_tests/resources</code> 디렉터리에 대한 별칭을 설정한다. 예를 들어 HTTP 테스트에서는 <code>src="/js-test-resources/js-test.js"</code>를 사용해 테스트 프레임워크에 접근할 수 있다.</p>
<h3 id="팁">팁</h3><p>각 빌더에서 가장 최근 ~100개 빌드에서 테스트가 어떻게 되었는지 보려면 <a href="https://test-results.appspot.com/">https://test-results.appspot.com/</a> 를 확인하라(페이지가 정기적으로 업데이트되고 있는 한).</p>
<p>타임아웃은 종종 텍스트 불일치이기도 하다. 래퍼 스크립트가 content_shell이 끝낼 기회를 갖기 전에 종료하기 때문이다. 예외는 테스트가 제대로 로드를 마쳤지만, 어떻게든 완료되었음을 래퍼에 알리는 텍스트 조각을 출력하기 전에 멈추는 경우다.</p>
<p>테스트가 buildbot에서는 실패(또는 크래시, 타임아웃)하지만 로컬 머신에서는 통과하는 이유는 무엇일 수 있을까?</p>
<ul>
<li>테스트가 로컬에서는 끝나지만 느리다면, 대략 10초 이상 걸린다면, 그것이 봇에서 타임아웃으로 불리는 이유일 것이다.</li>
<li>그렇지 않다면 테스트 집합의 일부로 실행해 본다. 이 테스트보다 하나나 둘(또는 열 개) 앞의 테스트가 무언가를 오염시켜 이 테스트가 실패하게 만들 가능성이 있다.</li>
<li>로컬에서 일관되게 동작한다면, 환경이 봇의 환경과 비슷한지 확인한다(webkit_tests step의 stdio 상단을 보면 모든 환경 변수 등을 볼 수 있다).</li>
<li>그 어느 것도 도움이 되지 않고 봇 자체에 접근할 수 있다면, 그곳에 로그인하여 문제를 수동으로 재현할 수 있는지 확인해야 할 수도 있다.</li>
</ul>
<h3 id="devtools-테스트-디버깅하기">DevTools 테스트 디버깅하기</h3><ul>
<li>다음 중 하나를 수행한다.<ul>
<li>옵션 A) <code>chromium/src</code> 폴더에서 실행:<br /><code>third_party/blink/tools/run_web_tests.py --additional-driver-flag='--remote-debugging-port=9222' --additional-driver-flag='--remote-allow-origins=*' --additional-driver-flag='--debug-devtools' --timeout-ms=6000000</code></li>
<li>옵션 B) http/tests/inspector 테스트를 디버깅해야 한다면, 위에서 설명한 대로 httpd를 시작한다. 그런 다음 content_shell을 실행한다.<br /><code>out/Default/content_shell --remote-debugging-port=9222 --additional-driver-flag='--remote-allow-origins=*' --additional-driver-flag='--debug-devtools' --run-web-tests http://127.0.0.1:8000/path/to/test.html</code></li>
</ul>
</li>
<li>stable/beta/canary Chrome에서 <code>http://localhost:9222</code>를 열고, 단일 링크를 클릭해 테스트가 로드된 devtools를 연다.</li>
<li>로드된 devtools에서 필요한 브레이크포인트를 설정하고 콘솔에서 <code>test()</code>를 실행해 실제로 테스트를 시작한다.</li>
</ul>
<p>NOTE: 테스트가 html 파일이라면 이는 레거시 테스트라는 의미이므로 다음을 추가해야 한다.</p>
<ul>
<li><p>테스트 코드에 다음과 같이 <code>window.debugTest = true;</code>를 추가한다.</p>
<pre><code class="hljs language-javascript"><span class="hljs-variable language_">window</span>.<span class="hljs-property">debugTest</span> = <span class="hljs-literal">true</span>;
<span class="hljs-keyword">function</span> <span class="hljs-title function_">test</span>(<span class="hljs-params"></span>) {
  <span class="hljs-comment">/* TEST CODE */</span>
}</code></pre></li>
</ul>
<h3 id="flaky-inspector-protocol-테스트-재현하기">flaky inspector protocol 테스트 재현하기</h3><p><a href="https://crrev.com/c/5318502">https://crrev.com/c/5318502</a> 는 inspector-protocol 테스트를 위한 로깅을 구현했다. 이 CL을 사용하면 각 테스트의 stderr에서 테스트와 브라우저가 주고받은 Chrome DevTools Protocol 메시지를 볼 수 있어야 한다.</p>
<p>이 로그를 사용해 실패나 타임아웃을 로컬에서 재현할 수 있다.</p>
<ul>
<li><p>로그 파일을 준비하고 각 줄이 JSON 형식의 프로토콜 메시지 하나를 포함하도록 한다. 원본 로그에서 prefix나 비프로토콜 메시지를 제거한다.</p>
</li>
<li><p>로컬 테스트 파일 버전이 로그 파일을 생성한 버전과 일치하는지 확인한다.</p>
</li>
<li><p>로그 파일을 사용해 테스트를 실행한다.</p>
<pre><code class="hljs language-sh">third_party/blink/tools/run_web_tests.py -t Release \
 --additional-driver-flag=<span class="hljs-string">"--inspector-protocol-log=/path/to/log.txt"</span> \
 http/tests/inspector-protocol/network/url-fragment.js</code></pre></li>
</ul>
<h2 id="회귀-이등분하기">회귀 이등분하기</h2><p><a href="https://git-scm.com/docs/git-bisect"><code>git bisect</code></a>를 사용하면 어떤 커밋이 웹 테스트를 깨뜨렸는지(또는 고쳤는지!) 완전히 자동화된 방식으로 찾을 수 있다. 미리 빌드된 Chromium 바이너리를 다운로드하는 <a href="http://dev.chromium.org/developers/bisect-builds-py">bisect-builds.py</a>와 달리, <code>git bisect</code>는 로컬 체크아웃에서 동작하므로 <code>content_shell</code>로 테스트를 실행할 수 있다.</p>
<p>이등분에는 몇 시간이 걸릴 수 있지만, 완전히 자동화되어 있으므로 밤새 실행해 두고 다음 날 결과를 볼 수 있다.</p>
<p>웹 테스트 회귀의 자동 이등분을 설정하려면 다음과 같은 스크립트를 만든다.</p>
<pre><code class="hljs language-bash"><span class="hljs-meta">#!/bin/bash</span>

<span class="hljs-comment"># 종료 코드 125는 git bisect에 해당 리비전을 건너뛰라고 알린다.</span>
gclient <span class="hljs-built_in">sync</span> || <span class="hljs-built_in">exit</span> 125
autoninja -C out/Debug -j100 blink_tests || <span class="hljs-built_in">exit</span> 125

third_party/blink/tools/run_web_tests.py -t Debug \
  --no-show-results --no-retry-failures \
  path/to/web/test.html</code></pre><p><code>out</code> 디렉터리, ninja args, 테스트 이름을 적절히 수정하고 스크립트를 <code>~/checkrev.sh</code>에 저장한다. 그런 다음 다음을 실행한다.</p>
<pre><code class="hljs language-bash"><span class="hljs-built_in">chmod</span> u+x ~/checkrev.sh  <span class="hljs-comment"># 스크립트를 실행 가능으로 표시한다</span>
git bisect start &lt;badrev&gt; &lt;goodrev&gt;
git bisect run ~/checkrev.sh
git bisect reset  <span class="hljs-comment"># bisect 세션을 종료한다</span></code></pre><h2 id="웹-테스트-리베이스라인하기">웹 테스트 리베이스라인하기</h2><p><a href="https://chromium.googlesource.com/chromium/src/+/main/docs/testing/web_test_expectations.md#How-to-rebaseline">How to rebaseline</a>을 참고하라.</p>

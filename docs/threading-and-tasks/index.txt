16:T107b8,<blockquote>
<p>이 문서는 <strong>Threading and Tasks in Chrome</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/threading_and_tasks.md"><code>docs/threading_and_tasks.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p>참고: 더 많은 예제는 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/threading_and_tasks_faq.md">스레딩과 태스크 FAQ</a>를 보라.</p>
<h2 id="개요">개요</h2><p>Chrome은 <a href="https://www.chromium.org/developers/design-documents/multi-process-architecture">멀티 프로세스 아키텍처</a>를 가지고 있으며, 각 프로세스는 매우 많이 멀티스레드화되어 있다. 이 문서에서는 각 프로세스가 공유하는 기본 스레딩 시스템을 살펴본다. 우리의 주된 목표는 브라우저를 매우 반응성 있게 유지하는 것이다. 지연 시간이나 워크로드에 대한 외부 요구사항이 없다면, Chrome은 <a href="https://stackoverflow.com/questions/1050222/what-is-the-difference-between-concurrency-and-parallelism#:~:text=Concurrency%20is%20when%20two%20or,e.g.%2C%20on%20a%20multicore%20processor.">높은 동시성을 갖지만 반드시 병렬적일 필요는 없는</a> 시스템이 되려고 한다.</p>
<p>Chromium이 동시성을 다루는 방식, 특히 시퀀스(Sequences)에 대한 기본 소개는 <a href="https://docs.google.com/presentation/d/1ujV8LjIUyPBmULzdT2aT9Izte8PDwbJi">여기</a>에서 볼 수 있다.</p>
<p>이 문서는 컴퓨터 과학의 <a href="https://en.wikipedia.org/wiki/Thread_(computing)">스레딩 개념</a>에 익숙하다고 가정한다.</p>
<h3 id="빠른-시작-가이드">빠른 시작 가이드</h3><ul>
<li>메인 스레드, 즉 브라우저 프로세스의 “UI” 스레드나 IO 스레드, 즉 IPC 수신을 위한 각 프로세스의 스레드에서 비용이 큰 계산이나 블로킹 IO를 수행하지 말라. 바쁜 UI / IO 스레드는 사용자에게 보이는 지연 시간을 유발할 수 있으므로, 그런 작업은 <a href="#%EC%8A%A4%EB%A0%88%EB%93%9C-%ED%92%80%EC%97%90-%EC%A7%81%EC%A0%91-%EA%B2%8C%EC%8B%9C">스레드 풀</a>에서 실행하는 것을 선호하라.</li>
<li>별도 스레드나 시퀀스에서 메모리의 같은 위치를 읽고/쓰는 일은 항상 피하라. 이는 <a href="https://en.wikipedia.org/wiki/Race_condition#Data_race">데이터 레이스</a>로 이어진다! 대신 시퀀스 사이에서 메시지를 전달하는 것을 선호하라. 락 사용 같은 메시지 전달의 대안은 권장되지 않는다.</li>
<li>서로 다른 시퀀스에 존재하는 여러 객체를 조율해야 한다면, 객체 수명에 주의하라.<ul>
<li>우발적인 데이터 레이스를 막기 위해 대부분의 클래스는 단일 시퀀스에서만 독점적으로 사용되도록 하는 것을 선호하라. 이 제약을 강제하는 데 도움이 되도록 <a href="https://source.chromium.org/chromium/chromium/src/+/main:base/sequence_checker.h">SEQUENCE_CHECKER</a>나 <a href="https://source.chromium.org/chromium/chromium/src/+/main:base/threading/sequence_bound.h">base::SequenceBound</a> 같은 유틸리티를 사용해야 한다.</li>
<li>경험칙으로 <a href="https://source.chromium.org/chromium/chromium/src/+/main:base/functional/bind.h;l=169;drc=ef1375f2c9fffa0d9cd664b43b0035c09fb70e99">base::Unretained</a>는 피하라. 보통 <a href="https://source.chromium.org/chromium/chromium/src/+/main:base/memory/weak_ptr.h">weak pointer</a>로 대체할 수 있다.</li>
<li><code>std::unique_ptr</code>를 통한 명시적 소유권을 선호한다.</li>
<li><a href="https://source.chromium.org/chromium/chromium/src/+/main:base/memory/scoped_refptr.h">scoped_refptr</a>은 여러 시퀀스에 걸쳐 여러 소유자가 있는 객체에 사용할 수 있다. 이는 보통 잘못된 설계 패턴이며 새 코드에는 권장되지 않는다.</li>
</ul>
</li>
</ul>
<h2 id="핵심-개념">핵심 개념</h2><ul>
<li><strong>태스크(Task)</strong>: 처리될 작업의 단위. 사실상 선택적으로 연결된 상태를 가진 함수 포인터다. Chrome에서는 각각 <code>base::BindOnce</code>와 <code>base::BindRepeating</code>으로 생성되는 <code>base::OnceCallback</code>과 <code>base::RepeatingCallback</code>이다. (<a href="https://chromium.googlesource.com/chromium/src/+/HEAD/docs/callback.md">문서</a>).</li>
<li><strong>태스크 큐(Task queue)</strong>: 처리될 태스크들의 큐.</li>
<li><strong>물리 스레드(Physical thread)</strong>: 운영체제가 제공하는 스레드, 예를 들어 POSIX의 pthread 또는 Windows의 CreateThread(). Chrome의 크로스 플랫폼 추상화는 <code>base::PlatformThread</code>다. 이것을 직접 사용할 일은 거의 없어야 한다.</li>
<li><strong><code>base::Thread</code></strong>: Quit()될 때까지 전용 태스크 큐에서 메시지를 계속 처리하는 물리 스레드. 직접 <code>base::Thread</code>를 만드는 일은 거의 없어야 한다.</li>
<li><strong>스레드 풀(Thread pool)</strong>: 공유 태스크 큐를 가진 물리 스레드들의 풀. Chrome에서는 <code>base::ThreadPoolInstance</code>다. Chrome 프로세스마다 정확히 하나의 인스턴스가 있고, <a href="https://cs.chromium.org/chromium/src/base/task/thread_pool.h"><code>base/task/thread_pool.h</code></a>를 통해 게시된 태스크를 처리한다. 따라서 <code>base::ThreadPoolInstance</code> API를 직접 사용할 필요는 드물다. 태스크 게시에 대해서는 뒤에서 더 설명한다.</li>
<li><strong>시퀀스(Sequence)</strong> 또는 <strong>가상 스레드(Virtual thread)</strong>: Chrome이 관리하는 실행 스레드. 물리 스레드처럼 주어진 시퀀스 / 가상 스레드에서는 어느 순간에도 하나의 태스크만 실행될 수 있으며, 각 태스크는 앞선 태스크들의 부수 효과를 본다. 태스크들은 순차적으로 실행되지만 각 태스크 사이에서 물리 스레드를 옮겨 다닐 수 있다.</li>
<li><strong>태스크 러너(Task runner)</strong>: 태스크를 게시할 수 있는 인터페이스. Chrome에서는 <code>base::TaskRunner</code>다.</li>
<li><strong>시퀀스 태스크 러너(Sequenced task runner)</strong>: 게시된 태스크가 게시 순서대로 순차 실행됨을 보장하는 태스크 러너. 각 태스크는 자기 앞 태스크의 부수 효과를 보도록 보장된다. 시퀀스 태스크 러너에 게시된 태스크는 보통 단일 스레드, 즉 가상 또는 물리 스레드에서 처리된다. Chrome에서는 <code>base::TaskRunner</code>의 일종인 <code>base::SequencedTaskRunner</code>다.</li>
<li><strong>단일 스레드 태스크 러너(Single-thread task runner)</strong>: 모든 태스크가 같은 물리 스레드에서 처리됨을 보장하는 시퀀스 태스크 러너. Chrome에서는 <code>base::SequencedTaskRunner</code>의 일종인 <code>base::SingleThreadTaskRunner</code>다. 가능할 때마다 우리는 <a href="#%EB%AC%BC%EB%A6%AC-%EC%8A%A4%EB%A0%88%EB%93%9C%EB%B3%B4%EB%8B%A4-%EC%8B%9C%ED%80%80%EC%8A%A4%EB%A5%BC-%EC%84%A0%ED%98%B8%ED%95%98%EB%9D%BC">스레드보다 시퀀스를 선호</a>한다.</li>
</ul>
<h2 id="스레딩-어휘">스레딩 어휘</h2><p>독자 참고: 다음 용어들은 일반적인 스레딩 명명법과 Chrome에서 사용하는 방식을 이어 주려는 시도다. 막 시작했다면 다소 무거울 수 있다. 이해하기 어렵다면 아래의 더 상세한 섹션으로 건너뛰고 필요할 때 다시 참조하는 것을 고려하라.</p>
<ul>
<li><strong>스레드-불안전(Thread-unsafe)</strong>: Chrome의 압도적 다수 타입은 의도적으로 스레드-불안전하다. 이런 타입/메서드에 대한 접근은 외부에서 동기화되어야 한다. 보통 스레드-불안전 타입은 그 상태에 접근하는 모든 태스크가 같은 <code>base::SequencedTaskRunner</code>에 게시되어야 하며, 디버그 빌드에서는 <code>SEQUENCE_CHECKER</code> 멤버로 이를 검증한다. 접근 동기화를 위해 락도 선택지일 수 있지만 Chrome에서는 <a href="#%EB%9D%BD-%EB%8C%80%EC%8B%A0-%EC%8B%9C%ED%80%80%EC%8A%A4-%EC%82%AC%EC%9A%A9">락보다 시퀀스를 선호</a>한다.</li>
<li><strong>스레드-친화(Thread-affine)</strong>: 이런 타입/메서드는 항상 같은 물리 스레드, 즉 같은 <code>base::SingleThreadTaskRunner</code>에서 접근되어야 하며, 보통 이를 검증하기 위한 <code>THREAD_CHECKER</code> 멤버를 가진다. 서드파티 API를 사용하거나 스레드-친화적인 리프 의존성이 있는 경우가 아니라면, Chrome에서 타입이 스레드-친화적이어야 할 이유는 거의 없다. <code>base::SingleThreadTaskRunner</code>는 <code>base::SequencedTaskRunner</code>의 일종이므로 스레드-친화는 스레드-불안전의 부분집합이다. 스레드-친화는 때때로 <strong>thread-hostile</strong>이라고도 불린다.</li>
<li><strong>스레드-안전(Thread-safe)</strong>: 이런 타입/메서드는 병렬로 안전하게 접근할 수 있다.</li>
<li><strong>스레드-호환(Thread-compatible)</strong>: 이런 타입은 const 메서드에 대한 안전한 병렬 접근을 제공하지만 non-const 접근 또는 const/non-const 혼합 접근에는 동기화가 필요하다. Chrome은 reader-writer 락을 노출하지 않는다. 따라서 이 용도는 스레드-안전한 방식으로 한 번 초기화되고, 그 이후 영원히 불변인 객체, 보통 전역 객체뿐이다. 초기화는 시작 단계의 단일 스레드 구간에서 하거나 <code>base::NoDestructor</code> 같은 스레드-안전 static-local-initialization 패러다임을 통해 지연 수행될 수 있다.</li>
<li><strong>불변(Immutable)</strong>: 생성 이후 수정될 수 없는 스레드-호환 타입의 부분집합.</li>
<li><strong>시퀀스-친화(Sequence-friendly)</strong>: 이런 타입/메서드는 <code>base::SequencedTaskRunner</code>에서 호출되는 것을 지원하는 스레드-불안전 타입이다. 이상적으로는 모든 스레드-불안전 타입이 그래야 하지만, 레거시 코드에는 단순한 스레드-불안전 시나리오에서도 스레드-친화성을 강제하는 과도한 체크가 때때로 있다. 자세한 내용은 아래 <a href="#%EB%AC%BC%EB%A6%AC-%EC%8A%A4%EB%A0%88%EB%93%9C%EB%B3%B4%EB%8B%A4-%EC%8B%9C%ED%80%80%EC%8A%A4%EB%A5%BC-%EC%84%A0%ED%98%B8%ED%95%98%EB%9D%BC">물리 스레드보다 시퀀스를 선호</a>를 보라.</li>
</ul>
<h3 id="스레드">스레드</h3><p>모든 Chrome 프로세스에는 다음이 있다.</p>
<ul>
<li>메인 스레드<ul>
<li>브라우저 프로세스에서는 BrowserThread::UI, 또는 iOS에서는 web::WebThread::UI: UI를 업데이트한다.</li>
<li>렌더러 프로세스에서는 Blink 메인 스레드: Blink의 대부분을 실행한다.</li>
</ul>
</li>
<li>IO 스레드<ul>

<li>더 일반적으로 대부분의 비동기 I/O는 이 스레드에서 일어난다. 예: base::FileDescriptorWatcher를 통해.</li>
<li>브라우저 프로세스에서는 BrowserThread::IO라고 부르며, iOS에서는 web::WebThread::IO라고 부른다.</li>
</ul>
</li>
<li>몇 개의 추가 특수 목적 스레드</li>
<li>그리고 범용 스레드 풀</li>
</ul>
<p>대부분의 스레드는 큐에서 태스크를 가져와 실행하는 루프를 가진다. 이 큐는 여러 스레드 사이에서 공유될 수 있다.</p>
<h3 id="태스크">태스크</h3><p>태스크는 비동기 실행을 위해 큐에 추가된 <code>base::OnceClosure</code>다.</p>
<p><code>base::OnceClosure</code>는 함수 포인터와 인자를 저장한다. 바인딩된 인자를 사용해 함수 포인터를 호출하는 <code>Run()</code> 메서드를 가진다. 이는 <code>base::BindOnce</code>를 사용해 생성된다. (<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/callback.md">Callback&lt;&gt; 및 Bind() 문서</a> 참고).</p>
<pre><code class="hljs language-plaintext">void TaskA() {}
void TaskB(int v) {}

auto task_a = base::BindOnce(&amp;TaskA);
auto task_b = base::BindOnce(&amp;TaskB, 42);</code></pre><p>태스크 그룹은 다음 방식 중 하나로 실행될 수 있다.</p>
<ul>
<li><a href="#%EB%B3%91%EB%A0%AC-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">병렬</a>: 태스크 실행 순서가 없으며, 어떤 스레드에서든 모두 동시에 실행될 수 있음</li>
<li><a href="#%EC%8B%9C%ED%80%80%EC%8A%A4-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">시퀀스</a>: 태스크가 게시 순서대로, 한 번에 하나씩, 어떤 스레드에서든 실행됨.</li>
<li><a href="#%EA%B0%99%EC%9D%80-%EC%8A%A4%EB%A0%88%EB%93%9C%EC%97%90-%EC%97%AC%EB%9F%AC-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">단일 스레드</a>: 태스크가 게시 순서대로, 한 번에 하나씩, 단일 스레드에서 실행됨.<ul>
<li><a href="#com-single-thread-apartmentsta-%EC%8A%A4%EB%A0%88%EB%93%9C%EC%97%90-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9Cwindows">COM 단일 스레드</a>: COM이 초기화된 단일 스레드 방식의 변형.</li>
</ul>
</li>
</ul>
<h3 id="물리-스레드보다-시퀀스를-선호하라">물리 스레드보다 시퀀스를 선호하라</h3><p>시퀀스 실행, 즉 가상 스레드 위 실행은 단일 스레드 실행, 즉 물리 스레드 위 실행보다 강하게 선호된다. 메인 스레드(UI)나 IO 스레드에 묶인 타입/메서드를 제외하면, 스레드 안전성은 직접 물리 스레드를 관리하는 것보다 <code>base::SequencedTaskRunner</code>를 통해 더 잘 달성된다. 아래 <a href="#%EC%8B%9C%ED%80%80%EC%8A%A4-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">시퀀스 태스크 게시</a>를 참고하라.</p>
<p>"현재 물리 스레드"용으로 노출된 모든 API에는 "현재 시퀀스"용 동등 API가 있다(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/threading_and_tasks_faq.md#How-to-migrate-from-SingleThreadTaskRunner-to-SequencedTaskRunner">매핑</a>).</p>
<p>시퀀스-친화 타입을 작성하는 중 리프 의존성에서 스레드-친화성 체크, 예를 들어 <code>THREAD_CHECKER</code>에 실패한다면, 그 의존성도 시퀀스-친화적으로 만드는 것을 고려하라. Chrome의 대부분 핵심 API는 시퀀스-친화적이지만, 일부 레거시 타입은 "현재 시퀀스"에 의존할 수 있고 더 이상 스레드-친화적일 필요가 없는데도 여전히 ThreadChecker/SingleThreadTaskRunner를 과도하게 사용할 수 있다.</p>
<h2 id="병렬-태스크-게시">병렬 태스크 게시</h2><h3 id="스레드-풀에-직접-게시">스레드 풀에 직접 게시</h3><p>어떤 스레드에서든 실행될 수 있고 다른 태스크와 순서나 상호 배제 요구사항이 없는 태스크는 <a href="https://cs.chromium.org/chromium/src/base/task/thread_pool.h"><code>base/task/thread_pool.h</code></a>에 정의된 <code>base::ThreadPool::PostTask*()</code> 함수 중 하나를 사용해 게시해야 한다.</p>
<pre><code class="hljs language-cpp">base::ThreadPool::<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;Task));</code></pre><p>이는 기본 traits로 태스크를 게시한다.</p>
<p><code>base::ThreadPool::PostTask*()</code> 함수는 호출자가 TaskTraits를 통해 태스크에 대한 추가 세부 정보를 제공할 수 있게 한다(<a href="#tasktraits%EB%A1%9C-%ED%83%9C%EC%8A%A4%ED%81%AC-%EC%A3%BC%EC%84%9D-%EB%8B%AC%EA%B8%B0">TaskTraits로 태스크 주석 달기</a> 참고).</p>
<pre><code class="hljs language-cpp">base::ThreadPool::<span class="hljs-built_in">PostTask</span>(
    FROM_HERE, {base::TaskPriority::BEST_EFFORT, <span class="hljs-built_in">MayBlock</span>()},
    base::<span class="hljs-built_in">BindOnce</span>(&amp;Task));</code></pre><h3 id="taskrunner를-통한-게시">TaskRunner를 통한 게시</h3><p>병렬 <a href="https://cs.chromium.org/chromium/src/base/task/task_runner.h"><code>base::TaskRunner</code></a>는 <code>base::ThreadPool::PostTask*()</code>를 직접 호출하는 것의 대안이다. 이는 주로 태스크가 병렬로, 시퀀스로, 또는 단일 스레드로 게시될지 미리 알 수 없을 때 유용하다(<a href="#%EC%8B%9C%ED%80%80%EC%8A%A4-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">시퀀스 태스크 게시</a>, <a href="#%EA%B0%99%EC%9D%80-%EC%8A%A4%EB%A0%88%EB%93%9C%EC%97%90-%EC%97%AC%EB%9F%AC-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">같은 스레드에 여러 태스크 게시</a> 참고). <code>base::TaskRunner</code>는 <code>base::SequencedTaskRunner</code>와 <code>base::SingleThreadTaskRunner</code>의 기반 클래스이므로, <code>scoped_refptr&lt;TaskRunner&gt;</code> 멤버는 <code>base::TaskRunner</code>, <code>base::SequencedTaskRunner</code>, 또는 <code>base::SingleThreadTaskRunner</code>를 담을 수 있다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-keyword">class</span> <span class="hljs-title class_">A</span> {
 <span class="hljs-keyword">public</span>:
  <span class="hljs-built_in">A</span>() = <span class="hljs-keyword">default</span>;

  <span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">PostSomething</span><span class="hljs-params">()</span> </span>{
    task_runner_-&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;A, &amp;DoSomething));
  }

  <span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">DoSomething</span><span class="hljs-params">()</span> </span>{
  }

 <span class="hljs-keyword">private</span>:
  scoped_refptr&lt;base::TaskRunner&gt; task_runner_ =
      base::ThreadPool::<span class="hljs-built_in">CreateTaskRunner</span>({base::TaskPriority::USER_VISIBLE});
};</code></pre><p>테스트가 태스크 실행 방식을 정확히 제어해야 하는 경우가 아니라면, <code>base::ThreadPool::PostTask*()</code>를 직접 호출하는 것을 선호한다. 테스트에서 태스크를 덜 침습적으로 제어하는 방법은 <a href="#%ED%85%8C%EC%8A%A4%ED%8A%B8">테스트</a>를 참고하라.</p>
<h2 id="시퀀스-태스크-게시">시퀀스 태스크 게시</h2><p>시퀀스는 게시 순서대로 한 번에 하나씩 실행되는 태스크들의 집합이다. 반드시 같은 스레드에서 실행되는 것은 아니다. 시퀀스의 일부로 태스크를 게시하려면 <a href="https://cs.chromium.org/chromium/src/base/task/sequenced_task_runner.h"><code>base::SequencedTaskRunner</code></a>를 사용하라.</p>
<h3 id="새-시퀀스에-게시">새 시퀀스에 게시</h3><p><code>base::SequencedTaskRunner</code>는 <code>base::ThreadPool::CreateSequencedTaskRunner()</code>로 생성할 수 있다.</p>
<pre><code class="hljs language-cpp">scoped_refptr&lt;SequencedTaskRunner&gt; sequenced_task_runner =
    base::ThreadPool::<span class="hljs-built_in">CreateSequencedTaskRunner</span>(...);

<span class="hljs-comment">// TaskB는 TaskA가 완료된 뒤 실행된다.</span>
sequenced_task_runner-&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;TaskA));
sequenced_task_runner-&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;TaskB));</code></pre><h3 id="현재-가상-스레드에-게시">현재 (가상) 스레드에 게시</h3><p>현재 (가상) 스레드에 게시하는 선호 방식은 <code>base::SequencedTaskRunner::GetCurrentDefault()</code>를 통하는 것이다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// 태스크는 현재 (가상) 스레드의 기본 태스크 큐에서 실행된다.</span>
base::SequencedTaskRunner::<span class="hljs-built_in">GetCurrentDefault</span>()-&gt;<span class="hljs-built_in">PostTask</span>(
    FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;Task));</code></pre><p><code>SequencedTaskRunner::GetCurrentDefault()</code>는 현재 가상 스레드의 기본 큐를 반환한다는 점에 유의하라. 여러 태스크 큐를 가진 스레드, 예를 들어 BrowserThread::UI에서는 이것이 현재 태스크가 속한 큐와 다를 수 있다. "현재" 태스크 러너는 의도적으로 static getter를 통해 노출되지 않는다. 이미 알고 있다면 직접 게시할 수 있고, 모른다면 유일하게 합리적인 대상은 기본 큐다. 자세한 논의는 <a href="https://bit.ly/3JvCLsX">https://bit.ly/3JvCLsX</a> 를 보라.</p>
<h2 id="락-대신-시퀀스-사용">락 대신 시퀀스 사용</h2><p>Chrome에서 락 사용은 권장되지 않는다. 시퀀스는 본질적으로 스레드 안전성을 제공한다. 락으로 직접 스레드 안전성을 관리하기보다 항상 같은 시퀀스에서 접근되는 클래스를 선호하라.</p>
<p><strong>스레드-안전하지만 스레드-친화적이지 않다; 어떻게 가능한가?</strong> 같은 시퀀스에 게시된 태스크들은 순차 순서로 실행된다. 시퀀스 태스크가 완료된 뒤 다음 태스크는 다른 워커 스레드가 가져갈 수 있지만, 그 태스크는 자신의 시퀀스에서 이전 태스크들이 일으킨 모든 부수 효과를 보도록 보장된다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-keyword">class</span> <span class="hljs-title class_">A</span> {
 <span class="hljs-keyword">public</span>:

<p>논블로킹 코드를 작성하기 위해 Chrome의 많은 API는 비동기적이다. 보통 이는 특정 스레드/시퀀스에서 실행되어야 하고 커스텀 delegate 인터페이스를 통해 결과를 반환하거나, 요청된 작업이 완료될 때 호출되는 <code>base::OnceCallback&lt;&gt;</code> 또는 <code>base::RepeatingCallback&lt;&gt;</code> 객체를 받는다는 의미다. 특정 스레드/시퀀스에서 작업을 실행하는 것은 위의 PostTask 섹션에서 다룬다.</p>
<h2 id="같은-스레드에-여러-태스크-게시">같은 스레드에 여러 태스크 게시</h2><p>여러 태스크가 같은 스레드에서 실행되어야 한다면 <a href="https://cs.chromium.org/chromium/src/base/task/single_thread_task_runner.h"><code>base::SingleThreadTaskRunner</code></a>에 게시하라. 같은 <code>base::SingleThreadTaskRunner</code>에 게시된 모든 태스크는 같은 스레드에서 게시 순서대로 실행된다.</p>
<h3 id="브라우저-프로세스의-메인-스레드-또는-io-스레드에-게시">브라우저 프로세스의 메인 스레드 또는 IO 스레드에 게시</h3><p>메인 스레드 또는 IO 스레드에 태스크를 게시하려면 <a href="https://cs.chromium.org/chromium/src/content/public/browser/browser_thread.h"><code>content/public/browser/browser_thread.h</code></a>의 <code>content::GetUIThreadTaskRunner({})</code> 또는 <code>content::GetIOThreadTaskRunner({})</code>를 사용하라.</p>
<p>이 메서드들에 추가 BrowserTaskTraits를 매개변수로 제공할 수 있지만, BrowserThreads에서는 여전히 일반적으로 드물며 고급 사용 사례에 한정해야 한다.</p>
<p>이전의 traits가 있는 base API에서 벗어나는 진행 중인 마이그레이션(<a href="https://docs.google.com/document/d/1tssusPykvx3g0gvbvU4HxGyn3MjJlIylnsH13-Tv6s4/edit?ts=5de99a52#heading=h.ss4tw38hvh3s">task APIs v3</a>)이 있으며, 코드베이스 전반에서 아직 다음 형태를 볼 수 있다. 이는 동등하다.</p>
<pre><code class="hljs language-cpp">base::<span class="hljs-built_in">PostTask</span>(FROM_HERE, {content::BrowserThread::UI}, ...);

base::<span class="hljs-built_in">CreateSingleThreadTaskRunner</span>({content::BrowserThread::IO})
    -&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, ...);</code></pre><p>참고: 마이그레이션 기간 동안 browser_thread.h API를 사용하려면 안타깝게도 계속해서 <a href="https://cs.chromium.org/chromium/src/content/public/browser/browser_task_traits.h"><code>content/public/browser/browser_task_traits.h</code></a>를 수동으로 include해야 한다.</p>
<p>메인 스레드와 IO 스레드는 이미 매우 바쁘다. 따라서 가능할 때는 범용 스레드에 게시하는 것을 선호하라(<a href="#%EB%B3%91%EB%A0%AC-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">병렬 태스크 게시</a>, <a href="#%EC%8B%9C%ED%80%80%EC%8A%A4-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">시퀀스 태스크 게시</a> 참고). 메인 스레드에 게시할 좋은 이유는 UI를 업데이트하거나 그것에 묶인 객체, 예를 들어 <code>Profile</code>에 접근하는 것이다. IO 스레드에 게시할 좋은 이유는 그것에 묶인 컴포넌트 내부, 예를 들어 IPC나 네트워크에 접근하는 것이다. 참고: IPC를 송수신하거나 네트워크에서 데이터를 송수신하기 위해 IO 스레드로 명시적인 post task를 할 필요는 없다.</p>
<h3 id="렌더러-프로세스의-메인-스레드에-게시">렌더러 프로세스의 메인 스레드에 게시</h3><p>TODO(blink-dev)</p>

<h3 id="현재-스레드에-게시">현재 스레드에 게시</h3><p>*** note<br /><strong>중요:</strong> 현재 태스크 시퀀스와 상호 배제가 필요하지만 반드시 현재 물리 스레드에서 실행될 필요는 없는 태스크를 게시하려면 <code>base::SingleThreadTaskRunner::GetCurrentDefault()</code> 대신 <code>base::SequencedTaskRunner::GetCurrentDefault()</code>를 사용하라(<a href="#%ED%98%84%EC%9E%AC-%EA%B0%80%EC%83%81-%EC%8A%A4%EB%A0%88%EB%93%9C%EC%97%90-%EA%B2%8C%EC%8B%9C">현재 시퀀스에 게시</a> 참고). 그러면 게시된 태스크의 요구사항을 더 잘 문서화하고, API를 불필요하게 물리 스레드-친화적으로 만드는 일을 피할 수 있다. 단일 스레드 태스크에서 <code>base::SequencedTaskRunner::GetCurrentDefault()</code>는 <code>base::SingleThreadTaskRunner::GetCurrentDefault()</code>와 동등하다.</p>
<hr />
<p>그럼에도 현재 물리 스레드에 태스크를 게시해야 한다면 <a href="https://source.chromium.org/chromium/chromium/src/+/main:base/task/single_thread_task_runner.h"><code>base::SingleThreadTaskRunner::CurrentDefaultHandle</code></a>을 사용하라.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// 태스크는 미래에 현재 스레드에서 실행된다.</span>

    FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;Task));</code></pre><h2 id="com-single-thread-apartmentsta-스레드에-태스크-게시windows">COM Single-Thread Apartment(STA) 스레드에 태스크 게시(Windows)</h2><p>COM Single-Thread Apartment(STA) 스레드에서 실행되어야 하는 태스크는 <code>base::ThreadPool::CreateCOMSTATaskRunner()</code>가 반환하는 <code>base::SingleThreadTaskRunner</code>에 게시해야 한다. <a href="#%EA%B0%99%EC%9D%80-%EC%8A%A4%EB%A0%88%EB%93%9C%EC%97%90-%EC%97%AC%EB%9F%AC-%ED%83%9C%EC%8A%A4%ED%81%AC-%EA%B2%8C%EC%8B%9C">같은 스레드에 여러 태스크 게시</a>에서 언급했듯, 같은 <code>base::SingleThreadTaskRunner</code>에 게시된 모든 태스크는 같은 스레드에서 게시 순서대로 실행된다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// Task(A|B|C)UsingCOMSTA는 같은 COM STA 스레드에서 실행된다.</span>

<span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">TaskAUsingCOMSTA</span><span class="hljs-params">()</span> </span>{

  <span class="hljs-comment">// ...</span>

<span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">TaskCUsingCOMSTA</span><span class="hljs-params">()</span> </span>{ }

<span class="hljs-keyword">auto</span> com_sta_task_runner = base::ThreadPool::<span class="hljs-built_in">CreateCOMSTATaskRunner</span>(...);
com_sta_task_runner-&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;TaskAUsingCOMSTA));
com_sta_task_runner-&gt;<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(&amp;TaskBUsingCOMSTA));</code></pre><h2 id="게시된-태스크의-메모리-순서-보장">게시된 태스크의 메모리 순서 보장</h2><p>이 태스크 시스템은 태스크를 게시하기 전 순차 실행의 모든 메모리 효과가 태스크가 실행을 시작할 때 <em>보이도록</em> 보장한다. 더 형식적으로는, <code>PostTask()</code> 호출과 게시된 태스크의 실행은 서로 <a href="https://preshing.com/20130702/the-happens-before-relation/">happens-before 관계</a>에 있다. 이는 <code>PostTaskAndReply()</code>를 포함해 <code>::base</code>에서 태스크를 게시하는 모든 변형에 대해 참이다. 마찬가지로 같은 SequencedTaskRunner의 일부로 시퀀스에서 실행되는 태스크들 사이에도 happens-before 관계가 존재한다.</p>
<p>이 보장은 알아 둘 필요가 있다. Chrome 태스크는 <code>base::OnceCallback</code> 안으로 복사된 즉각적인 데이터 너머의 메모리에 흔히 접근하며, 이 happens-before 관계 덕분에 태스크 자체 안에서 추가 동기화를 피할 수 있기 때문이다. 매우 구체적인 예로, 태스크를 게시하는 스레드에서 막 초기화한 메모리에 대한 포인터를 바인딩하는 콜백을 생각해 보라.</p>
<p>더 제약된 모델도 주목할 만하다. 실행은 서로 다른 태스크 러너에서 실행되는 태스크들로 나뉠 수 있으며, 각 태스크는 명시적 동기화 없이 메모리의 특정 객체에 <em>독점적으로</em> 접근한다. 다른 태스크를 게시하는 것은 이 '소유권'(객체들의 소유권)을 다음 태스크로 이전한다. 이를 통해 객체 소유권 개념을 종종 태스크 러너 수준까지 확장할 수 있고, 이는 추론에 유용한 불변 조건을 제공한다. 이 모델은 락과 atomic 연산을 피하면서도 race condition을 피할 수 있게 한다. 단순성 때문에 이 모델은 Chrome에서 흔히 사용된다.</p>
<h2 id="tasktraits로-태스크-주석-달기">TaskTraits로 태스크 주석 달기</h2><p><a href="https://cs.chromium.org/chromium/src/base/task/task_traits.h"><code>base::TaskTraits</code></a>는 스레드 풀이 더 나은 스케줄링 결정을 내리는 데 도움이 되는 태스크 정보를 캡슐화한다.</p>
<p><code>base::TaskTraits</code>를 받는 메서드에는 기본 traits로 충분할 때 <code>{}</code>를 전달할 수 있다. 기본 traits는 다음 태스크에 적합하다.</p>
<ul>
<li>블로킹하지 않음(MayBlock과 WithBaseSyncPrimitives 참고);</li>
<li>사용자 차단 활동과 관련됨;<br />(명시적으로 또는 그런 컴포넌트와의 순서 의존성을 가짐으로써 암묵적으로)</li>
<li>종료를 막거나 종료 시 건너뛸 수 있음(스레드 풀이 적합한 기본값을 자유롭게 선택함).</li>
</ul>
<p>이 설명과 맞지 않는 태스크는 명시적 TaskTraits와 함께 게시되어야 한다.</p>
<p><a href="https://cs.chromium.org/chromium/src/base/task/task_traits.h"><code>base/task/task_traits.h</code></a>는 사용 가능한 traits에 대한 포괄적 문서를 제공한다. content 계층도 BrowserThread에 태스크를 게시하기 쉽게 하기 위해 <a href="https://cs.chromium.org/chromium/src/content/public/browser/browser_task_traits.h"><code>content/public/browser/browser_task_traits.h</code></a>에 추가 traits를 제공한다.</p>
<p>아래는 <code>base::TaskTraits</code>를 지정하는 몇 가지 예다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// 이 태스크에는 명시적 TaskTraits가 없다. 블로킹할 수 없다. 우선순위는</span>
<span class="hljs-comment">// USER_BLOCKING이다. 종료를 막거나 종료 시 건너뛰어진다.</span>
base::ThreadPool::<span class="hljs-built_in">PostTask</span>(FROM_HERE, base::<span class="hljs-built_in">BindOnce</span>(...));

<span class="hljs-comment">// 이 태스크는 가장 높은 우선순위를 가진다. 스레드 풀은</span>
<span class="hljs-comment">// USER_VISIBLE 및 BEST_EFFORT 태스크보다 먼저 이를 스케줄한다.</span>

    FROM_HERE, {base::TaskPriority::USER_BLOCKING},
    base::<span class="hljs-built_in">BindOnce</span>(...));

<span class="hljs-comment">// 이 태스크는 가장 낮은 우선순위를 가지며 블로킹이 허용된다. 예를 들어</span>
<span class="hljs-comment">// 디스크에서 파일을 읽을 수 있다.</span>
base::ThreadPool::<span class="hljs-built_in">PostTask</span>(
    FROM_HERE, {base::TaskPriority::BEST_EFFORT, base::<span class="hljs-built_in">MayBlock</span>()},
    base::<span class="hljs-built_in">BindOnce</span>(...));

<ul>
<li>각 작업 항목마다 <code>PostTask()</code>를 호출하여 상당한 오버헤드를 발생시키는 경우.</li>
<li>작업을 분할하는 고정 개수의 <code>PostTask()</code> 호출이 오랫동안 실행될 수 있는 경우. 많은 컴포넌트가 “코어 수”만큼의 태스크를 게시하고 모두가 모든 코어를 사용하기를 기대할 때 문제가 된다. 이런 경우 스케줄러는 동일 우선순위의 여러 요청에 공정하게 대응할 맥락 및/또는 높은 우선순위 작업이 들어올 때 낮은 우선순위 작업이 양보하도록 요청할 능력이 부족하다.</li>
</ul>
<p>완전한 예제는 <a href="https://cs.chromium.org/chromium/src/base/task/job_perftest.cc"><code>base/task/job_perftest.cc</code></a>를 보라.</p>

<h3 id="실행-중인-job에-추가-작업-더하기">실행 중인 job에 추가 작업 더하기</h3><p>새 작업 항목이 추가되고 API 사용자가 추가 스레드가 워커 태스크를 병렬로 호출하기를 원한다면, 최대 동시성이 증가한 직후 <code>JobHandle/JobDelegate::NotifyConcurrencyIncrease()</code>가 호출되어야 <em>한다</em>.</p>
<h2 id="테스트">테스트</h2><p>자세한 내용은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/threading_and_tasks_testing.md">태스크를 게시하는 컴포넌트 테스트</a>를 보라.</p>
<p><code>base::SingleThreadTaskRunner::CurrentDefaultHandle</code>, <code>base::SequencedTaskRunner::CurrentDefaultHandle</code> 또는 <a href="https://cs.chromium.org/chromium/src/base/task/thread_pool.h"><code>base/task/thread_pool.h</code></a>의 함수를 사용하는 코드를 테스트하려면, 테스트 범위에 대해 <a href="https://cs.chromium.org/chromium/src/base/test/task_environment.h"><code>base::test::TaskEnvironment</code></a>를 인스턴스화하라. BrowserThreads가 필요하다면 <code>base::test::TaskEnvironment</code> 대신 <code>content::BrowserTaskEnvironment</code>를 사용하라.</p>
<p>테스트는 <code>base::RunLoop</code>를 사용해 <code>base::test::TaskEnvironment</code>의 message pump를 실행할 수 있다. 이는 <code>Quit()</code>될 때까지, 명시적으로 또는 <code>RunLoop::QuitClosure()</code>를 통해 실행되게 할 수 있고, 또는 준비된 태스크에 대해 <code>RunUntilIdle()</code>을 실행한 뒤 즉시 반환하게 할 수 있다.</p>
<p>TaskEnvironment는 TestTimeouts::action_timeout() 이후 명시적으로 quit되지 않았다면 RunLoop::Run()이 GTEST_FAIL()하도록 구성한다. 이는 테스트 대상 코드가 RunLoop를 quit하도록 트리거하지 못했을 때 테스트가 멈춰 있는 것보다 낫다. 제한 시간은 base::test::ScopedRunLoopTimeout으로 재정의할 수 있다.</p>

  <span class="hljs-comment">// bit.ly/run-until-idle-with-care2.</span>
  task_environment_.<span class="hljs-built_in">RunUntilIdle</span>();
  <span class="hljs-comment">// E, H, I가 실행되었다.</span>
}</code></pre><h2 id="새-프로세스에서-threadpool-사용">새 프로세스에서 ThreadPool 사용</h2><p><a href="https://cs.chromium.org/chromium/src/base/task/thread_pool.h"><code>base/task/thread_pool.h</code></a>의 함수를 사용하려면 프로세스에서 ThreadPoolInstance가 먼저 초기화되어야 한다. Chrome 브라우저 프로세스와 자식 프로세스(렌더러, GPU, 유틸리티)에서 ThreadPoolInstance 초기화는 이미 처리되어 있다. 다른 프로세스에서 ThreadPoolInstance를 사용하려면 main 함수 초기에 ThreadPoolInstance를 초기화하라.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// 이것은 기본 params로 ThreadPoolInstance를 초기화하고 시작한다.</span>
base::ThreadPoolInstance::<span class="hljs-built_in">CreateAndStartWithDefaultParams</span>(<span class="hljs-string">"process_name"</span>);
<span class="hljs-comment">// 이제 base::ThreadPool trait와 함께 base/task/thread_pool.h API를 사용할 수 있다.</span>
<span class="hljs-comment">// 태스크는 게시되는 대로 스케줄된다.</span>

<span class="hljs-comment">// 이것은 ThreadPoolInstance를 초기화한다.</span>
base::ThreadPoolInstance::<span class="hljs-built_in">Create</span>(<span class="hljs-string">"process_name"</span>);
<span class="hljs-comment">// 이제 base::ThreadPool trait와 함께 base/task/thread_pool.h API를 사용할 수 있다. Start()가</span>
<span class="hljs-comment">// 호출되기 전까지 스레드는 생성되지 않고 태스크도 스케줄되지 않는다.</span>
base::ThreadPoolInstance::<span class="hljs-built_in">Get</span>()-&gt;<span class="hljs-built_in">Start</span>(params);
<span class="hljs-comment">// 이제 ThreadPool은 스레드를 생성하고 태스크를 스케줄할 수 있다.</span></code></pre><p>그리고 main 함수 후반에 ThreadPoolInstance를 종료하라.</p>
<pre><code class="hljs language-cpp">base::ThreadPoolInstance::<span class="hljs-built_in">Get</span>()-&gt;<span class="hljs-built_in">Shutdown</span>();
<span class="hljs-comment">// TaskShutdownBehavior::BLOCK_SHUTDOWN으로 게시된 태스크와</span>
<span class="hljs-comment">// Shutdown() 호출 전에 실행을 시작한 TaskShutdownBehavior::SKIP_ON_SHUTDOWN 태스크는</span>
<span class="hljs-comment">// 이제 실행을 완료했다. TaskShutdownBehavior::CONTINUE_ON_SHUTDOWN으로 게시된 태스크는</span>
<span class="hljs-comment">// 아직 실행 중일 수 있다.</span></code></pre><h2 id="taskrunner-소유권의존성-주입을-권장하지-않음">TaskRunner 소유권(의존성 주입을 권장하지 않음)</h2><p>TaskRunner는 여러 컴포넌트를 거쳐 전달되어서는 안 된다. 대신 TaskRunner를 사용하는 컴포넌트가 그것을 생성하는 주체여야 한다.</p>
<p>TaskRunner가 많은 컴포넌트를 거쳐 전달되다가 결국 리프에서만 사용되었던 리팩터링의 <a href="https://codereview.chromium.org/2885173002/">이 예</a>를 보라. 이제 리프는 <a href="https://cs.chromium.org/chromium/src/base/task/thread_pool.h"><code>base/task/thread_pool.h</code></a>에서 직접 TaskRunner를 얻을 수 있고 그래야 한다.</p>
<p>위에서 언급했듯 <code>base::test::TaskEnvironment</code>는 단위 테스트가 하위 TaskRunner에서 게시된 태스크를 제어할 수 있게 한다. 테스트가 태스크 순서를 더 정밀하게 제어해야 하는 드문 경우에는 TaskRunner의 의존성 주입이 유용할 수 있다. 그런 경우 선호되는 접근 방식은 다음과 같다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-keyword">class</span> <span class="hljs-title class_">Foo</span> {
 <span class="hljs-keyword">public</span>:

  <span class="hljs-comment">// 테스트에서 |background_task_runner_|를 재정의한다.</span>
  <span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">SetBackgroundTaskRunnerForTesting</span><span class="hljs-params">(
      scoped_refptr&lt;base::SequencedTaskRunner&gt; background_task_runner)</span> </span>{
    background_task_runner_ = std::<span class="hljs-built_in">move</span>(background_task_runner);
  }

 <span class="hljs-keyword">private</span>:
  scoped_refptr&lt;base::SequencedTaskRunner&gt; background_task_runner_ =
      base::ThreadPool::<span class="hljs-built_in">CreateSequencedTaskRunner</span>(
          {base::<span class="hljs-built_in">MayBlock</span>(), base::TaskPriority::BEST_EFFORT});
}</code></pre><p>이 방식은 단위 테스트가 리프 계층을 직접 사용할 것이므로 //chrome과 해당 컴포넌트 사이의 모든 배관 계층을 제거하는 것을 여전히 가능하게 한다는 점에 유의하라.</p>
<h2 id="faq">FAQ</h2><p>더 많은 예제는 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/threading_and_tasks_faq.md">스레딩과 태스크 FAQ</a>를 보라.</p>
<h2 id="내부-구조">내부 구조</h2><h3 id="sequencemanager">SequenceManager</h3><p><a href="https://cs.chromium.org/chromium/src/base/task/sequence_manager/sequence_manager.h">SequenceManager</a>는 서로 다른 속성, 예를 들어 우선순위나 공통 태스크 유형을 가진 TaskQueue들을 관리하며, 게시된 모든 태스크를 하나의 backing sequence로 multiplex한다. 이는 보통 MessagePump다. 사용되는 message pump의 유형에 따라 UI 메시지 같은 다른 이벤트도 처리될 수 있다. Windows에서는 시간이 허락하는 한 APC 호출과 등록된 HANDLE 집합으로 전송된 signal도 처리될 수 있다.</p>
<h3 id="messagepump">MessagePump</h3><p><a href="https://cs.chromium.org/chromium/src/base/message_loop/message_pump.h">MessagePump</a>는 native message를 처리하고 주기적으로 delegate(SequenceManager)에 cycle을 제공할 책임이 있다. MessagePump는 delegate callback과 native message 처리를 섞어 어느 유형의 이벤트도 cycle을 굶주리지 않도록 신경 쓴다.</p>
<p>서로 다른 <a href="https://cs.chromium.org/chromium/src/base/message_loop/message_pump_type.h">MessagePumpTypes</a>가 있으며, 가장 일반적인 것은 다음과 같다.</p>
<ul>
<li><p>DEFAULT: 태스크와 타이머만 지원</p>
</li>
<li><p>UI: native UI 이벤트, 예를 들어 Windows message 지원</p>
</li>
<li><p>IO: 비동기 IO 지원(파일 I/O가 아니다!)</p>
</li>
<li><p>CUSTOM: 사용자가 제공한 MessagePump 인터페이스 구현</p>
</li>
</ul>
<h3 id="runloop">RunLoop</h3><p>RunLoop는 현재 스레드에 연결된 RunLoop::Delegate, 보통 SequenceManager를 실행하기 위한 헬퍼 클래스다. 스택에 RunLoop를 만들고 Run/Quit를 호출해 중첩 RunLoop를 실행하라. 하지만 production code에서는 중첩 loop를 피하라!</p>
<h3 id="태스크-재진입성">태스크 재진입성</h3><p>SequenceManager에는 태스크 재진입성 보호가 있다. 이는 태스크가 처리 중이라면 첫 번째 태스크가 끝날 때까지 두 번째 태스크가 시작될 수 없다는 뜻이다. 태스크를 처리하는 중 내부 message pump가 생성될 때 재진입이 일어날 수 있다. 그 내부 pump는 native message를 처리하고, 이 native message가 암묵적으로 내부 태스크를 시작할 수 있다. 내부 message pump는 dialog(DialogBox), common dialog(GetOpenFileName), OLE 함수(DoDragDrop), printer 함수(StartDoc) 및 <em>많은</em> 다른 것들에 의해 생성된다.</p>
<pre><code class="hljs language-cpp">내부 태스크 처리가 필요할 때의 샘플 우회 방법:
  HRESULT hr;
  {
    CurrentThread::ScopedAllowApplicationTasksInNativeNestedLoop allow;
    hr = <span class="hljs-built_in">DoDragDrop</span>(...); <span class="hljs-comment">// modal message loop를 암묵적으로 실행한다.</span>
  }
  <span class="hljs-comment">// |hr|을 처리한다(DoDragDrop()이 반환한 결과).</span></code></pre><p>CurrentThread::ScopedAllowApplicationTasksInNativeNestedLoop를 사용하기 전에, 당신의 태스크가 재진입 가능(nestable)하고 모든 전역 변수가 안정적이며 접근 가능함을 반드시 확인하라.</p>
<h2 id="일반-사용을-위한-api">일반 사용을 위한 API</h2><p>사용자 코드는 SequenceManager API에 직접 접근할 필요가 거의 없어야 한다. 이 API들은 스케줄링을 다루는 코드를 위한 것이기 때문이다. 대신 다음을 사용해야 한다.</p>
<ul>
<li><p>base::RunLoop: SequenceManager가 바인딩된 스레드에서 이를 구동한다.</p>
</li>
<li><p>base::Thread/SequencedTaskRunner::CurrentDefaultHandle: 그 위에서 실행 중인 태스크에서 SequenceManager TaskQueue로 다시 게시한다.</p>
</li>
<li><p>SequenceLocalStorageSlot : 외부 상태를 시퀀스에 바인딩한다.</p>
</li>
<li><p>base::CurrentThread : 현재 스레드에 바인딩된 Task 관련 API의 부분집합에 대한 proxy</p>
</li>
<li><p>Embedder는 특정 loop에 태스크를 게시하기 위한 자체 static accessor를 제공할 수 있다. 예: content::BrowserThreads.</p>
</li>
</ul>
<h3 id="singlethreadtaskexecutor와-taskenvironment">SingleThreadTaskExecutor와 TaskEnvironment</h3><p>SequenceManager와 TaskQueue를 직접 다루는 대신, 단순한 태스크 게시 환경, 즉 하나의 기본 태스크 큐가 필요한 코드는 <a href="https://cs.chromium.org/chromium/src/base/task/single_thread_task_executor.h">SingleThreadTaskExecutor</a>를 사용할 수 있다.</p>
<p>단위 테스트는 매우 구성 가능한 <a href="https://cs.chromium.org/chromium/src/base/test/task_environment.h">TaskEnvironment</a>를 사용할 수 있다.</p>
<h2 id="messageloop와-messageloopcurrent">MessageLoop와 MessageLoopCurrent</h2><p>코드나 문서에서 MessageLoop 또는 MessageLoopCurrent에 대한 참조를 마주칠 수 있다. 이 클래스들은 더 이상 존재하지 않으며, 우리는 이에 대한 모든 참조를 제거하는 과정에 있다. <code>base::MessageLoopCurrent</code>는 <code>base::CurrentThread</code>로 대체되었고, <code>base::MessageLoop</code>의 drop-in 대체물은 <code>base::SingleThreadTaskExecutor</code>와 <code>base::Test::TaskEnvironment</code>다.</p>

15:T2a63,<blockquote>
<p>이 문서는 <strong>Configuration: Prefs, Settings, Features, Switches &amp; Flags</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/configuration.md"><code>docs/configuration.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p>이 문서는 Chromium의 모든 런타임 구성 표면을 개괄하고, 각각에 대한 적절한 사용처와 표준 패턴을 논의한다. 이들 중 일부는 사용자가 쓰도록 의도되었고, 일부는 개발자가, 일부는 시스템 관리자가 쓰도록 의도되었다.</p>
<h2 id="prefs">Prefs</h2><p>예: prefs::kAllowDinosaurEasterEgg, 일명 "allow_dinosaur_easter_egg"</p>
<p>Prefs는 중앙 pref 서비스에 등록해서 구현하며, 보통 <a href="https://source.chromium.org/chromium/chromium/src/+/main:chrome/browser/profiles/profile.h;l=189;drc=b0378e4b67a5dbdb15acf0341ccd51acda81c8e0">Profile::RegisterProfilePrefs</a>를 통해 등록한다. Prefs는 타입이 지정된 값을 저장하며, 이 값은 재시작 후에도 유지되고 Sync 서비스를 통해 브라우저 인스턴스 간에 동기화될 수 있다. pref 저장소는 여러 종류가 있으며, <a href="https://www.chromium.org/developers/design-documents/preferences">prefs 문서</a>에 자세히 문서화되어 있다. Prefs는 엔터프라이즈 정책을 통해 직접 구성할 수 있다.</p>
<p>Prefs:</p>
<ul>
<li>사용자에게 직접 노출되지는 <em>않는다</em></li>
<li>사용자의 언어로 현지화되지는 <em>않는다</em></li>
<li>해당 pref에 대응하는 정책이 있으면 엔터프라이즈 정책으로 구성할 수 <em>있다</em><br />(임의의 pref를 설정할 수 있게 해 주는 포괄 정책은 없다)</li>
<li>사용 중일 때 UMA로 보고되지는 <em>않는다</em></li>
<li>chrome://version에 포함되지는 <em>않는다</em></li>
<li>재시작 후에도 자동으로 지속된다(보통은)</li>
</ul>
<h2 id="features">Features</h2><p>예: base::kDCheckIsFatalFeature</p>
<p>Features는 어디서든 <a href="https://cs.chromium.org/chromium/src/base/feature_list.h?sq=package:chromium&amp;g=0&amp;l=53">base::Feature</a>를 만들어 구현한다. Features는 서버 측 실험을 통해 활성화할 수도 있고, 명령줄에서 "--enable-features"를 사용해 활성화할 수도 있다. 어떤 features가 사용 중인지는 UMA 메트릭으로 추적되며, chrome://version의 "Variations" 필드에서 볼 수 있다. 단, 릴리스 빌드에서는 variation의 문자열 이름 대신 일련의 해시만 chrome://version에 표시되지만, 필요한 경우 이 해시를 다시 문자열 이름으로 바꿀 수 있다. 이는 Google 내부 도구를 참조해서 수행한다.</p>
<p><em>Features는 런타임 조건부 동작을 추가하는 가장 좋은 방법이다.</em></p>
<p>Features:</p>
<ul>
<li>사용자에게 직접 노출되지는 <em>않는다</em></li>
<li>사용자의 언어로 현지화되지는 <em>않는다</em></li>
<li>엔터프라이즈 정책으로 구성할 수는 <em>없다</em></li>
<li>사용 중일 때 UMA/crash로 보고된다</li>
<li>chrome://version에 포함된다</li>
<li>재시작 후 자동으로 지속되지는 <em>않는다</em></li>
</ul>
<h2 id="switches">Switches</h2><p>예: switches::kIncognito, 일명 "--incognito"</p>
<p>Switches는 코드베이스 어디서든 <a href="https://cs.chromium.org/chromium/src/base/command_line.h?type=cs&amp;l=98">base::CommandLine::ForCurrentProcess</a> 안에 특정 switch가 있는지 또는 그 값이 무엇인지를 검사해서 구현한다. switch의 중앙 등록소는 없으며, 사실상 어떤 목적에든 사용할 수 있다.</p>
<p>Switches:</p>
<ul>
<li>사용자에게 직접 노출되지는 <em>않는다</em></li>
<li>사용자의 언어로 현지화되지는 <em>않는다</em></li>
<li>엔터프라이즈 정책으로 구성할 수는 <em>없다</em>(Chrome OS에서는 FeatureFlagsProto를 통해 예외)</li>
<li>사용 중일 때 UMA로 보고되지는 <em>않는다</em></li>
<li>chrome://version에 포함된다</li>
<li>재시작 후 자동으로 지속되지는 <em>않는다</em></li>
</ul>
<p>일반적으로 switches는 base::Feature를 사용하는 것보다 열등하다. base::Feature는 같은 기능과 낮은 엔지니어링 오버헤드를 제공하면서도 UMA 보고와 연결되기 때문이다. 새 코드는 switches 대신 base::Feature를 사용해야 한다. 단, 구성 값이 문자열인 경우는 예외인데, features는 임의의 문자열 값을 받을 수 없기 때문이다.</p>
<h2 id="flags">Flags</h2><p>예: chrome://flags/#ignore-gpu-blocklist</p>
<p>Flags는 flag를 설명하는 항목을 <a href="https://cs.chromium.org/chromium/src/chrome/browser/about_flags.cc">about_flags.cc</a>에 추가하고, <a href="https://cs.chromium.org/chromium/src/chrome/browser/flag-metadata.json">flag-metadata</a>에도 메타데이터를 추가해서 구현한다. Flags에는 이름과 설명이 있으며, chrome://flags에 표시된다. Flags에는 만료 milestone도 있는데, 이 milestone이 지나면 해당 UI에서 숨겨지고 비활성화된 뒤, 나중에 제거된다. Flags는 feature 또는 switches 집합으로 뒷받침되며, flag 값에 따라 브라우저 시작 시 이들을 활성화한다.</p>
<p>Flags는 보통 feature의 출시 전 테스트를 가능하게 하기 위한 임시 수단이어야 한다. 영구 flags(만료가 -1인 것)는 다음 중 하나에 해당할 때만 사용해야 한다.</p>
<ul>
<li>지원/디버깅 목적으로 정기적으로 사용되는 flag인 경우. 예를 들어 사용자가 가능한 문제를 제거해 보기 위해 켜 보도록 요청받는 경우(예: ignore-gpu-blocklist)</li>
<li>명령줄 switches를 사용할 수 없는 환경(예: 모바일)에서 지속적인 QA/테스트 목적으로 사용되는 flag인 경우</li>
</ul>
<p>"사용자가 이 기능을 켜거나 꺼야 할 수도 있다"는 영구 flag의 충분한 정당화가 아니다. 가능하다면 사용자가 기능을 끄고 싶어 하거나 끌 필요가 없도록 기능을 설계해야 한다. 하지만 그 선택지를 유지해야 한다면, 번역과 지원이 있는 완전한 setting(아래 참고)으로 승격해야 한다. 반면 "개발자/QA가 이 기능을 켜거나 꺼야 할 수도 있다"는 영구 flag의 정당화가 된다.</p>
<p>Flags:</p>
<ul>
<li>사용자에게 직접 노출된다</li>
<li>사용자의 언어로 현지화되지는 <em>않는다</em></li>
<li>엔터프라이즈 정책으로 구성할 수는 <em>없다</em></li>
<li>사용 중일 때 UMA로 보고된다(<code>Launch.FlagsAtStartup</code>을 통해)</li>
<li>chrome://version에 포함되지는 <em>않는다</em></li>
<li>재시작 후 자동으로 지속된다</li>
</ul>
<h2 id="settings">Settings</h2><p>예: "Show home button"</p>
<p>Settings는 WebUI에서 구현되며, chrome://settings 또는 그 하위 페이지 중 하나에 표시된다. 보통 setting의 값을 저장하는 pref에 바인딩된다. Settings는 추가 비용이 비교적 큰 편이다. 현지화가 필요하고, chrome://settings 안에 어떻게 맞춰 넣을지 결정하기 위해 어느 정도의 UX 관여가 필요하며, 문서화와 지원 자료도 필요하기 때문이다. 많은 settings가 prefs를 통해 구현되지만, 모든 prefs가 settings에 대응하는 것은 아니다. 일부 prefs는 재시작 사이에 내부 브라우저 상태를 추적하는 데 사용된다.</p>
<p>Settings:</p>
<ul>
<li>사용자에게 직접 노출된다</li>
<li>사용자의 언어로 현지화된다</li>
<li>엔터프라이즈 정책으로 구성할 수는 <em>없다</em>(다만 이를 뒷받침하는 prefs는 가능할 수 있다)</li>
<li>사용 중일 때 UMA로 보고되지는 <em>않는다</em></li>
<li>chrome://version에 포함되지는 <em>않는다</em></li>
<li>재시작 후 자동으로 지속된다(이를 뒷받침하는 prefs를 통해)</li>
</ul>
<p>최종 사용자가 이 동작을 바꾸고 싶어 할 수 있다면 setting을 추가해야 한다. 어떤 것이 flag여야 하는지 setting이어야 하는지 판단하는 괜찮은 리트머스 테스트는 다음과 같다. "코드를 읽거나 쓸 수 없는 사람이 이것을 바꾸고 싶어 할 것인가?"</p>

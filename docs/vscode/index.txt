16:T95ab,<blockquote>
<p>이 문서는 <strong>Visual Studio Code Dev</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/vscode.md"><code>docs/vscode.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p><strong><a href="#%EC%84%A4%EC%A0%95">여기</a>에서 시작하세요</strong>.</p>
<p><a href="https://code.visualstudio.com">Visual Studio Code (VS Code)</a>는 Windows, macOS, Linux용 무료 오픈 소스 경량 고성능 코드 편집기로, <a href="https://www.electronjs.org/">Electron</a>/Chromium 기반입니다. JavaScript, TypeScript, Node.js에 대한 내장 지원이 있으며, intellisense, 디버깅, 구문 강조 등을 추가하는 풍부한 확장 생태계를 갖추고 있습니다. C++, Python, Go, Java 같은 많은 언어에서는 많은 설정 없이도 동작합니다.</p>
<p>Visual Studio 같은 완전한 IDE는 아닙니다. 두 제품은 완전히 별개의 제품입니다. Visual Studio와의 유일한 공통점은 둘 다 Microsoft 제품이라는 점입니다.</p>
<p>잘 동작하는 것들은 다음과 같습니다.</p>
<ul>
<li><strong>코드 편집</strong>은 특히 <a href="#%ED%82%A4%EB%B3%B4%EB%93%9C-%EB%8B%A8%EC%B6%95%ED%82%A4">키보드 단축키</a>에 익숙해지면 잘 동작합니다. VS Code는 반응성이 매우 좋고 Chromium 같은 큰 코드베이스도 처리할 수 있습니다.</li>
<li><strong>Git 통합</strong>은 매우 훌륭합니다. 내장된 나란히 보기, 로컬 커밋, 그리고 <a href="https://marketplace.visualstudio.com/items?itemName=donjayamanne.githistory">히스토리</a>와 <a href="https://marketplace.visualstudio.com/items?itemName=ryu1kn.annotator">blame 보기</a>를 위한 확장까지 제공합니다.</li>
<li><a href="https://code.visualstudio.com/Docs/editor/debugging"><strong>디버깅</strong></a>은 시작 시간이 꽤 길 수 있지만(Linux에서 gdb 사용 시 약 40초, Windows에서는 훨씬 짧음) 잘 동작합니다. 코드를 단계별로 실행하고, 변수를 검사하고, 여러 스레드의 호출 스택을 볼 수 있습니다.<ul>
<li>Python 코드 디버깅에 대한 자세한 내용은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/vscode_python.md">여기</a>를 참조하세요.</li>
</ul>
</li>
<li><strong>Command Palette</strong>는 파일 열기와 솔루션 검색을 매우 쉽게 해 줍니다.</li>
<li><strong>빌드</strong>도 잘 동작합니다. 빌드 도구를 쉽게 통합할 수 있습니다. 경고와 오류는 별도 페이지에 표시되며, 클릭해서 해당 코드 줄로 이동할 수 있습니다.</li>
<li><strong>VS Code Remote</strong>는 원격으로 호스팅되는 코드를 편집할 수 있게 해 주며, vscode-clangd 같은 계산 비용이 큰 플러그인도 원격 서버에서 실행할 수 있게 해 줍니다. 재택근무에 좋습니다. 자세한 내용은 <a href="#Remote">Remote 섹션</a>을 참조하세요.</li>
</ul>
<h2 id="이-페이지-업데이트하기">이 페이지 업데이트하기</h2><p>이 문서를 최신 상태로 유지해 주세요. VS Code는 아직 활발히 개발 중이며 변경될 수 있습니다. 이 문서는 Chromium git 저장소에 체크인되어 있으므로, 변경할 경우 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/documentation_guidelines.md">문서화 가이드라인</a>을 읽고 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/contributing.md">change list를 제출</a>하세요.</p>
<p>모든 파일 경로와 명령은 Linux와 macOS에서 테스트되었습니다. Windows에서는 약간 다른 설정이 필요할 수 있습니다. 그에 맞게 이 페이지를 업데이트해 주세요.</p>
<h2 id="설정">설정</h2><h3 id="설치">설치</h3><p>*** promo<br />Googler: 대신 <a href="http://go/vscode/install">go/vscode/install</a>을 참조하세요.</p>
<hr />
<p>개발 플랫폼에 맞는 적절한 버전을 설치하려면 <a href="https://code.visualstudio.com/docs/setup/setup-overview">Visual Studio Code 설정하기</a>의 단계를 따르세요.</p>
<h3 id="사용법">사용법</h3><p>Linux 또는 macOS에서 실행하려면:</p>
<pre><code class="hljs language-bash"><span class="hljs-built_in">cd</span> /path/to/chromium/src
code .</code></pre><p>Code Insiders를 설치했다면 바이너리 이름은 대신 <code>code-insiders</code>입니다.</p>
<p>VS Code에는 프로젝트 또는 솔루션 파일이 필요하지 않습니다. 하지만 기본 디렉터리(즉, 프로젝트 루트 폴더)의 <code>.vscode</code> 폴더에 워크스페이스 설정을 저장합니다. 자세한 내용은 <a href="#chromium%EC%9A%A9-%EC%84%A4%EC%A0%95">Chromium 워크스페이스 설정</a> 섹션을 참조하세요.</p>
<h3 id="유용한-확장">유용한 확장</h3><p>지금까지는 언어 지원이 많지 않은 기본 VS Code 버전을 갖고 있습니다. 다음으로 유용한 확장 몇 가지를 설치하겠습니다.</p>
<h4 id="권장-확장">권장 확장</h4><p>다음 확장들은 대부분 매일 사용하게 될 것입니다.</p>
<p>설치 방법은 2가지입니다.</p>
<ul>
<li><a href="#%EA%B6%8C%EC%9E%A5-%ED%99%95%EC%9E%A5-%EC%84%A4%EC%B9%98">권장 확장 설치</a>의 지침을 따릅니다.</li>
<li>수동 설치. 확장 창(<code>Ctrl+Shift+X</code>, macOS에서는 <code>Cmd+Shift+X</code>)으로 이동해 다음 확장 이름을 검색합니다.</li>
</ul>
<p>*** aside<br />참고: 아래에 언급된 모든 확장 설정은 이미 <code>tools/vscode/settings.json</code>에 설정되어 있습니다. 해당 파일을 워크스페이스로 복사하는 <a href="#chromium%EC%9A%A9-%EC%84%A4%EC%A0%95">지침</a>을 따랐다면 아무것도 할 필요가 없습니다.</p>
<hr />
<ul>
<li><a href="https://marketplace.visualstudio.com/items?itemName=Google.cros-ide"><strong>ChromiumIDE</strong></a> - 핵심 도구를 한곳에 고정해 Chromium/ChromiumOS 개발을 더 쉽고 빠르게 만드는 중요한 확장입니다.</li>
<li><a href="https://marketplace.visualstudio.com/items?itemName=solomonkinard.chromium-context"><strong>Chromium Context</strong></a> - 열린 파일에 대해 코드 소유자, 릴리스 버전, 작성자 blame 목록 같은 Chromium 전용 컨텍스트를 단일 탭에 제공합니다.</li>

<h3 id="javaandroid-지원">Java/Android 지원</h3><p>VSCode에서 Java/Android 지원을 얻기 위해 사용할 수 있는 확장은 두 가지입니다.</p>
<ul>
<li>a.<br /><a href="https://marketplace.visualstudio.com/items?itemName=Google.cros-ide">ChromiumIDE</a></li>
<li>b.<br /><a href="https://marketplace.visualstudio.com/items?itemName=redhat.java">Language Support for Java™ by Red Hat</a></li>
</ul>
<p>ChromiumIDE는 다른 확장보다 훨씬 빠르고 안정적입니다. 주된 이유는 백그라운드 인덱싱과 영구 캐시에 의존하지 않기 때문입니다. 이 확장은 GN에서 빌드 구성을 추출하는 방법을 알고 있으므로 명령줄에서 스크립트를 수동으로 실행하지 않아도 동작합니다. 반면, 다른 확장에 비해 기능은 제한적입니다(예: 디버거는 아직 지원되지 않음).</p>
<p>Language Support for Java™ by Red Hat은 Eclipse JDT Language Server 기반이므로 더 많은 기능을 갖고 있습니다. 하지만 매우 느린 것으로 알려져 있으며(VSCode의 요청을 처리하기 전에 단일 스레드에서 전체 프로젝트를 인덱싱하는 데 수십 분이 걸림), 안정성도 낮습니다(소스 checkout을 sync하면 종종 혼란스러워져 캐시를 지우고 다시 인덱싱을 기다려야 함).</p>
<h4 id="a-chromiumide">a. ChromiumIDE</h4><p>VSCode marketplace에서 <a href="https://marketplace.visualstudio.com/items?itemName=Google.cros-ide">ChromiumIDE</a>의 최신 <strong>pre-release</strong> 버전을 설치하세요. 확장 버전이 <strong>0.35.32</strong> 이상인지 확인하세요.</p>
<p>그런 다음 Chromium 소스 트리를 포함하는 VSCode 워크스페이스를 열고(하위 디렉터리를 열어도 괜찮음) Java 파일을 여세요. 아직 설정하지 않았다면 기본 빌드 출력 디렉터리(예: <code>out/Default</code>)를 선택하라는 메시지가 표시됩니다.</p>
<h4 id="b-language-support-for-java-by-red-hat">b. Language Support for Java™ by Red Hat</h4><ol>
<li><p><strong>VS Code 워크스페이스 <code>settings.json</code>에 다음을 추가하세요:</strong></p>
<pre><code class="hljs language-plaintext">"java.import.gradle.enabled": false,
"java.import.maven.enabled": false</code></pre><p>이렇게 하면 language server가 Chromium 소스 트리 어디에서든 찾을 수 있는 <em>모든</em> Gradle 및 Maven 프로젝트를 빌드하려고 시도하는 것을 막을 수 있으며, 이는 보통 큰 혼란을 초래합니다.</p>
<pre><code class="hljs language-plaintext">"java.jdt.ls.java.home": "&lt;&lt; ABSOLUTE PATH TO YOUR WORKING COPY OF CHROMIUM &gt;&gt;/src/third_party/jdk/current"</code></pre><p>이것은 선택 사항이지만, language server가 호스트 시스템의 임의 JDK가 아니라 Chromium 빌드 시스템과 같은 JDK를 사용하도록 보장해 문제 가능성을 줄입니다.</p>
<p>또한 Java Language Server가 사용할 수 있는 리소스를 늘리세요. 예:</p>
<pre><code class="hljs language-plaintext">"java.jdt.ls.vmargs": "-XX:+UseParallelGC -XX:GCTimeRatio=4 -XX:AdaptiveSizePolicyWeight=90 -Dsun.zip.disableMemoryMapping=true -Xmx64G -Xms100m -Xlog:disable"</code></pre></li>
<li><p><strong><a href="https://marketplace.visualstudio.com/items?itemName=redhat.java"><em>Language Support for Java™ by Red Hat</em></a> 확장을 설치하세요.</strong> 다른 확장은 필요하지 않습니다.</p>
</li>
<li><p>일반적인 방식(즉, gn과 ninja 명령 사용)으로 <strong>코드를 빌드하세요</strong>. 그러면 다음 단계에 필요한 빌드 구성 파일이 생성됩니다. 또한 자동 생성 코드가 language server에 보이게 됩니다.</p>
</li>
<li><p><code>src</code> 디렉터리에서 <code>build/android/generate_vscode_project.py</code>를 실행해 <strong>Eclipse JDT 프로젝트를 생성하세요</strong>. 예를 들어 빌드 출력 디렉터리가 <code>out/Debug-x86</code>이고 빌드 대상이 <code>//chrome/android:chrome_java</code>라면 다음을 실행합니다: <code>build/android/generate_vscode_project.py --output-dir out/Debug-x86 --build-config gen/chrome/android/chrome_java.build_config.json</code>. 이렇게 하면 <code>src</code> 디렉터리에 <code>.project</code>와 <code>.classpath</code>가 생성됩니다.</p>
</li>
<li><p>생성된 프로젝트 가져오기를 시작하도록 VS Code 창을 <strong>다시 로드</strong>하세요.</p>
</li>
<li><p><strong>Java 소스 파일을 연 다음 language server가 프로젝트를 빌드할 때까지 몇 분 기다리세요</strong>.</p>
</li>
<li><p><strong>완료!</strong> 이제 빌드에 포함된 모든 <code>.java</code> 파일에 대해 완전한 Java 언어 지원을 사용할 수 있어야 합니다.</p>
</li>
</ol>
<h5 id="알려진-문제">알려진 문제</h5><ul>
<li><code>GEN_JNI</code>와 관련된 오류는 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/jni_zero/README.md">자동 생성된</a> <code>GEN_JNI</code> 클래스의 여러 정의에 대해 language server가 (정당하게) 혼란스러워하기 때문에 발생합니다. 이는 JNI generator의 알려진 특이 사항입니다.</li>
</ul>
<h5 id="문제-해결">문제 해결</h5><ul>
<li>이전 지침에 따라 <code>generate_vscode_classpath.py</code>를 사용했거나 뭔가 잘못되었다고 생각한다면, command palette에서 <code>Java: Clean Java Language Server Workspace</code>를 실행해 language server의 내부 상태를 지워 보세요. 이렇게 하면 language server가 생성된 Eclipse JDT 프로젝트를 가져와 내부 워크스페이스를 다시 빌드하게 됩니다.</li>
</ul>
<h4 id="자동-포매팅">자동 포매팅</h4><p>Chromium의 Java 코드는 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/clang_format.md">clang-format</a>을 사용해 포맷됩니다. VS Code가 clang-format을 사용해 Java 파일을 포맷하도록 하려면 <a href="https://marketplace.visualstudio.com/items?itemName=xaver.clang-format"><em>Clang-Format</em> 확장</a>을 설치하고 워크스페이스 <code>settings.json</code>에서 Java의 기본 포매터로 설정하세요.</p>
<pre><code class="hljs language-json"><span class="hljs-attr">"[java]"</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">{</span>
  <span class="hljs-attr">"editor.defaultFormatter"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"xaver.clang-format"</span>
<span class="hljs-punctuation">}</span></code></pre><p>clang-format 버전 차이로 인한 잠재적인 포매팅 차이를 피하려면, 확장이 <code>git cl format</code>과 같은 방식으로 clang-format을 실행하도록 구성하는 것이 좋습니다. 워크스페이스 <code>settings.json</code>에 다음을 추가하면 됩니다.</p>
<pre><code class="hljs language-json"><span class="hljs-attr">"clang-format.executable"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"&lt;&lt; PATH TO YOUR CHROMIUM WORKING COPY &gt;&gt;/src/buildtools/linux64/clang-format"</span></code></pre><h2 id="chromium용-설정">Chromium용 설정</h2><p>VS Code는 JSON 파일을 통해 구성됩니다. 이 섹션에서는 더 나은 Chromium 개발 경험을 위해 VS Code를 구성하는 방법을 설명합니다.</p>
<p>*** aside<br />참고: VS Code 사용자 정의에 대한 더 일반적인 소개는 <a href="https://code.visualstudio.com/docs/customization/overview">VS Code 문서</a>를 참조하세요.</p>
<hr />
<p>Chromium 저장소에는 기본 구성이 일부 포함되어 있습니다. Chromium checkout에 대해 VS Code를 초기화하려면 다음 명령을 실행하세요.</p>
<pre><code class="hljs language-bash"><span class="hljs-built_in">cd</span> /path/to/chromium/src
<span class="hljs-built_in">mkdir</span> .vscode
<span class="hljs-built_in">cp</span> tools/vscode/*.json .vscode/
<span class="hljs-built_in">cp</span> tools/vscode/cpp.code-snippets .vscode/</code></pre><p>완료했다면 다음 섹션으로 진행해 권장 확장을 설치하고 사용자 정의를 수행하세요.</p>
<h3 id="권장-확장-설치">권장 확장 설치</h3><p><a href="#%EC%9C%A0%EC%9A%A9%ED%95%9C-%ED%99%95%EC%9E%A5">유용한 확장</a> 섹션에서 설명했듯이 Chromium 개발에 도움이 되는 필수 확장들이 있습니다. 아래 단계를 따르세요.</p>
<ol>
<li>VS Code의 Command Palette(<code>Ctrl+Shift+P</code>, macOS에서는 <code>Cmd+Shift+P</code>)에서 <code>Show Recommended Extensions</code>를 입력하고 <code>Enter</code>를 누릅니다.</li>
<li>EXTENSIONS 사이드바의 WORKSPACE RECOMMENDATIONS 섹션에서 <code>Install Workspace Recommended Extensions</code>(구름 아이콘으로 표시됨)를 클릭합니다.</li>
</ol>
<p>이제 모두 준비되었습니다.</p>
<h3 id="워크스페이스-설정-사용자-정의">워크스페이스 설정 사용자 정의</h3><p><a href="https://chromium.googlesource.com/chromium/src/+/main/tools/vscode/settings.json">//tools/vscode/settings.json</a> 파일을 열어 기본 설정을 확인하세요. 더 나은 팀 개발을 가능하게 하기 위해 추가 또는 제거한 설정을 자유롭게 커밋하거나, 개인 취향에 맞게 <code>.vscode/settings.json</code>에서 로컬 설정을 변경하세요.</p>
<p>*** aside<br />참고: 이 설정들은 워크스페이스 폴더(Explorer 탭에 표시되는 루트 폴더)가 Chromium의 <code>src/</code> 디렉터리라고 가정합니다. 그렇지 않은 경우 <code>${workspaceFolder}</code>에 대한 모든 참조를 <code>src/</code> 경로로 바꾸세요.</p>
<hr />
<h3 id="태스크">태스크</h3><p>다음으로 VS Code에 코드를 컴파일하고, 테스트를 실행하고, 빌드 출력에서 경고와 오류를 읽는 방법을 알려 줍니다.</p>
<p><code>.vscode/tasks.json</code> 파일을 여세요. 이 파일은 기본적인 작업을 수행하는 태스크를 제공합니다. 상황과 필요에 맞게 명령을 조정해야 할 수 있습니다. 예를 들어 대부분의 태스크를 실행하기 전에 해당 파일의 <code>chromeOutputDir</code> 값을 설정해야 합니다.</p>
<p>이제 <code>Ctrl+P</code>(macOS에서는 <code>Cmd+Shift+P</code>)를 사용하고 "task "와 원하는 숫자를 입력해 태스크를 실행할 수 있습니다. 빌드 태스크 중 하나를 선택하면 빌드 출력이 터미널 패널에 표시됩니다. <code>F8</code> / <code>Shift-F8</code>을 사용해 빌드 문제 사이를 빠르게 이동하세요. 태스크 실행에 대한 자세한 내용은 <a href="#%ED%83%9C%EC%8A%A4%ED%81%AC-%EC%9D%B4%EB%A6%84">태스크 이름</a>을 참조하세요.</p>
<p>intellisense를 활성화했지만 include 경로를 올바르게 설정하지 않았다면, 문제 사이를 이동할 때 찾을 수 없는 모든 include 파일로도 이동하려고 하여 많은 잡음이 생깁니다. include 경로를 고치거나 다음과 같이 intellisense를 "tag parser" 모드로 설정할 수 있습니다.</p>
<ol>
<li>Preferences(<code>Ctrl+Shift+P</code> &gt; "Preferences: Open User Settings")를 엽니다.</li>
<li>설정 검색 상자에 "intellisense engine"을 입력합니다.</li>
<li>제공자로 "Tag Parser"를 선택합니다.</li>
</ol>
<p>참고: Chromebook에서는 **🔍+&lt;맨 윗줄에서 ESC가 아닌 8번째 버튼&gt;**을 사용하세요. 대부분의 경우 이는 8 키 바로 위에 가장 가까운 맨 윗줄 버튼입니다.</p>
<h3 id="실행-명령">실행 명령</h3><p>실행 명령은 Visual Studio의 <code>F5</code>와 동일합니다. 어떤 프로그램이나 디버거를 실행합니다. 선택적으로 <code>tasks.json</code>에 정의된 태스크를 실행할 수도 있습니다. 실행 명령은 debug view(<code>Ctrl+Shift+D</code>)에서 실행할 수 있습니다.</p>
<p><code>.vscode/launch.json</code> 파일을 열고 예시 실행 명령을 상황과 필요에 맞게 조정하세요(예: "type" 값은 Windows용으로 조정해야 함).</p>
<h3 id="키-바인딩">키 바인딩</h3><p>키 바인딩을 편집하려면 <code>Ctrl+K, Ctrl+S</code>를 누르세요. 왼쪽에는 기본값이, 오른쪽에는 <code>.vscode/keybindings.json</code> 파일에 저장된 오버라이드가 표시됩니다. 살펴보고 상황과 필요에 맞게 조정하세요. 키 바인딩을 변경하려면 해당 키 바인딩을 오른쪽으로 복사하세요. 꽤 자명합니다.</p>
<p><code>CodeSearchOpen</code> 같은 확장에서 지정한 명령을 포함해 어떤 명령이든 키에 바인딩할 수 있습니다. 예를 들어 <code>CodeSearchOpen</code>을 <code>F2</code>에 바인딩하려면 <code>{ "key": "F2", "command": "cs.open" },</code>를 추가하면 됩니다. 명령 제목 <code>CodeSearchOpen</code>은 동작하지 않습니다. 확장의 <a href="https://github.com/chaopeng/vscode-chromium-codesearch/blob/master/package.json">package.json 파일</a>에서 실제 명령 이름을 가져와야 합니다.</p>
<p>다른 편집기에 익숙하다면 선호하는 keymap도 설치할 수 있습니다. 예를 들어 eclipse keymap을 설치하려면 <code>vscode-eclipse-keybindings</code> 확장을 설치하세요. 더 많은 keymap은 <a href="https://marketplace.visualstudio.com/search?target=vscode&amp;category=Keymaps">marketplace</a>에서 찾을 수 있습니다.</p>
<h3 id="알려진-문제에-대한-수정">알려진 문제에 대한 수정</h3><h4 id="windows의-git">Windows의 Git</h4><p>컴퓨터에 <code>depot_tools</code> Git만 설치되어 있고 PATH에 들어 있더라도, VS Code는 <code>git.exe</code>를 찾는 것처럼 보여 이를 무시합니다. Git 통합이 동작하도록 하려면 설정에 다음을 추가해야 합니다.</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  <span class="hljs-attr">"git.path"</span><span class="hljs-punctuation">:</span> <span class="hljs-string">"C:\\src\\depot_tools\\git.bat"</span>

  <span class="hljs-comment">// more settings here...</span>
<span class="hljs-punctuation">}</span></code></pre><p>팁: <code>Ctrl+Shift+P</code>를 사용하고 "Preferences: Open User Settings (JSON)" 동사를 사용해 설정 JSON 파일로 이동할 수 있습니다(어떤 이유에서인지 <code>git.path</code>를 폴더 설정으로 지정하는 것은 동작하지 않는 것 같습니다).</p>
<h3 id="remote">Remote</h3><p>*** promo<br />Googler: 대신 <a href="http://go/vscode-remote">go/vscode-remote</a>를 참조하세요.</p>
<hr />
<p>VS Code에는 이제 코드가 다른 곳에 호스팅되어 있는 동안 노트북에서 VS Code를 사용할 수 있게 하는 <a href="https://code.visualstudio.com/docs/remote/remote-overview">Remote</a> 프레임워크가 있습니다. 이는 clangd도 원격으로 실행할 수 있게 해 주는 vscode-clangd 플러그인과 함께 사용할 때 특히 빛을 발합니다.</p>
<p>이를 실행하려면 Remote pack 확장을 설치한 다음 ssh config 파일에 원격 연결이 있는지 확인하세요.</p>
<p><code>~/.ssh/config</code>:</p>
<pre><code class="hljs language-plaintext">Host my-connection
  HostName my-remote-host.corp.company.com</code></pre><p>그러면 VS Code가 왼쪽의 'Remote Explorer' 섹션에 이 연결을 나열합니다. 이 연결로 VS Code를 실행하려면 나열된 호스트 이름 옆의 '+window' 아이콘을 클릭하세요. 폴더를 선택하라고 나오면 'src' 폴더 루트를 사용하세요. 그러면 'Remote' 모드의 새 VS Code 창이 열립니다. <em><strong>이제 vscode-clangd 등과 같은 확장을 원격 연결에 대해 별도로 설치할 수 있습니다.</strong></em></p>
<h4 id="chromebook">Chromebook</h4><p>Googler의 경우, Crostini를 사용하지 않고 Chromebook에서 원격 개발을 설정하는 Google 전용 지침은 <a href="http://go/vscode/remote_development_via_web">여기</a>에 있습니다.</p>
<h4 id="windows-및-ssh">Windows 및 SSH</h4><p>VS Code 원격 도구에는 기본적으로 Windows에 설치되어 있지 않은 'sshd'가 필요합니다.</p>
<p>Googler의 경우 워크스테이션에 sshd가 이미 설치되어 있어야 하며, <a href="http://go/building-chrome-win">go/building-chrome-win</a>의 설정 지침을 따랐다면 VS Code가 원격으로 동작해야 합니다. 여전히 문제가 있다면 <a href="http://go/vscode-remote#windows">go/vscode-remote#windows</a>를 참조하세요.</p>
<p>비-Googler는 <a href="https://docs.microsoft.com/en-us/windows-server/administration/openssh/openssh_install_firstuse">OpenSSH 서버 설치</a>에 대한 Microsoft 지침을 따르면 됩니다. 이 단계를 따르면 VS Code가 원격으로 동작해야 합니다.</p>
<h3 id="스니펫">스니펫</h3><p><a href="https://chromium.googlesource.com/chromium/src/+/main/tools/vscode/cpp.code-snippets">//tools/vscode/cpp.code-snippets</a>에 유용한 스니펫이 제공되어 있으며, 이는 이미 <code>.vscode/cpp.code-snippets</code>로 워크스페이스에 설치되어 있습니다.</p>
<h3 id="팁">팁</h3><h4 id="out-폴더"><code>out</code> 폴더</h4><p>자동 생성 코드는 <code>out/</code>의 하위 폴더에 들어갑니다. 이는 이 파일들이 VS Code에서 무시되며(위 files.exclude 참조), 예를 들어 quick-open(<code>Ctrl+P</code>)에서 열 수 없다는 뜻입니다. 버전 1.21 기준으로 VS Code는 부정 glob 명령을 지원하지 않지만, <code>out/Debug/gen</code>만 포함하도록 제외 패턴 집합을 정의할 수 있습니다.</p>
<pre><code class="hljs language-plaintext">"files.exclude": {
  // Ignore build output folders. Except out/Debug/gen/
  "out/[^D]*/": true,
  "out/Debug/[^g]*": true,
  "out/Debug/g[^e]*": true,
  "out_*/**": true,
},</code></pre><p>지원되면, symlink 대신 files.exclude에서 다음을 사용할 수 있습니다.</p>
<pre><code class="hljs language-plaintext">"!out/Debug/gen/**": true</code></pre><h4 id="vs-code를-git-편집기로-사용하기">VS Code를 git 편집기로 사용하기</h4><p>git 커밋 메시지 등에 VS Code를 편집기로 사용하려면 <code>~/.gitconfig</code> 파일에 <code>[core] editor = "code --wait"</code>를 추가하세요. 편집기 시작이 nano나 vim보다 상당히 느리다는 점에 유의하세요. VS Code를 merge tool로 사용하려면 <code>[merge] tool = code</code>를 추가하세요.</p>
<h4 id="태스크-이름">태스크 이름</h4><p>태스크 이름을 <code>1-build_chrome_debug</code>, <code>2-build_chrome_release</code> 등으로 지정했다는 점에 유의하세요. 이렇게 하면 번호를 눌러 태스크를 빠르게 실행할 수 있습니다. <code>Ctrl+P</code>를 누르고 <code>task &lt;n&gt;</code>을 입력하세요. 여기서 <code>&lt;n&gt;</code>은 태스크 번호입니다. 태스크 실행을 위한 키보드 단축키를 만들 수도 있습니다. <code>File &gt; Preferences &gt; Keyboard Shortcuts</code>로 이동해 <code>{ "key": "ctrl+r", "command": "workbench.action.tasks.runTask", "when": "!inDebugMode" }</code>를 추가하세요. 그러면 <code>Ctrl+R</code>을 누르고 <code>&lt;n&gt;</code>을 입력하면 충분합니다.</p>
<h4 id="노트북에서-작업하기">노트북에서 작업하기</h4><p>배터리를 절약하려면 git status 자동 새로고침을 비활성화하는 것이 좋습니다.</p>
<pre><code class="hljs language-plaintext">"git.autorefresh": false,</code></pre><h4 id="여러-git-저장소에서-편집하기">여러 Git 저장소에서 편집하기</h4><p>Chromium 저장소의 일부인 여러 Git 저장소에서 자주 작업한다면, Chromium에 체크인된 <code>.gitignore</code> 파일에 포함된 폴더 아래에 있는 파일에 대해 내장 도구가 예상대로 동작하지 않는 것을 볼 수 있습니다.</p>
<p>이를 우회하려면 편집하는 디렉터리를 워크스페이스 구성의 별도 <code>folders</code> 항목으로 추가하고, Chromium에서 무시되는 디렉터리가 Chromium <code>src</code> 경로보다 <strong>앞에</strong> 나열되도록 하세요.</p>
<p>이를 편집하려면 <code>Settings</code> -&gt; <code>Workspace</code> 탭 선택으로 이동하고, JSON으로 열기(오른쪽 위 버튼)를 선택한 다음, 다음과 같이 <code>folders</code>를 구성하세요(로컬 설정과 사용 방식에 맞게 경로를 변경하세요).</p>
<pre><code class="hljs language-plaintext">{
  "folders": [
    {
      "path": "chromium/src/third_party/perfetto"
    },
    {
      "path": "chromium/src"
    }
  ]
}</code></pre><h3 id="linux에서-chromium을-디버깅할-때-file-resource-is-not-available을-열-수-없음">Linux에서 Chromium을 디버깅할 때 $File resource is not available을 열 수 없음</h3><p>Chromium은 <a href="https://docs.google.com/document/d/1OX4jY_bOCeNK7PNjVRuBQE9s6BQKS8XRNWGK8FEyh-E/edit?usp=sharing">최근</a> 파일 경로를 출력 디렉터리 기준 상대 경로로 변경했습니다. <code>strip_absolute_paths_from_debug_symbols</code>가 true인지(기본값) <code>gn args out/$dir --list</code>로 확인하고, true라면 <code>cwd</code>를 출력 디렉터리로 설정하세요. 그렇지 않으면 <code>cwd</code>를 <code>${workspaceFolder}</code>로 설정하세요.</p>

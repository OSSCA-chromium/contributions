<p>이 문서는 <strong>Linux Build Instructions</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/linux/build_instructions.md"><code>docs/linux/build_instructions.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p>다른 플랫폼에 대한 지침은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/get_the_code.md">코드 가져오기</a> 페이지에 링크되어 있습니다.</p>
<h2 id="google-직원을-위한-지침">Google 직원을 위한 지침</h2><p>Google 직원인가요? 대신 <a href="https://goto.google.com/building-chrome">go/building-chrome</a>을 보세요.</p>
<h2 id="시스템-요구사항">시스템 요구사항</h2><ul>
<li>최소 8GB RAM이 있는 x86-64 머신. 16GB 이상을 강력히 권장합니다. 머신에 SSD가 있다면 RAM이 각각 8GB/16GB인 머신에는 스왑을 &gt;=32GB/&gt;=16GB로 두는 것을 권장합니다.</li>
<li>최소 100GB의 여유 디스크 공간. 같은 드라이브에 있을 필요는 없습니다. 빌드용으로 HDD에 약 50~80GB를 할당하세요.</li>
<li>Git과 Python v3.9+가 이미 설치되어 있어야 합니다(그리고 <code>python3</code>는 Python v3.9+ 바이너리를 가리켜야 합니다). 시스템에 적절한 버전이 아직 없다면 Depot_tools가 <code>$depot_tools/python-bin</code>에 적절한 Python 버전을 번들로 제공합니다.</li>
<li>Chromium의 빌드 인프라와 <code>depot_tools</code>는 현재 Python 3.11을 사용합니다. 더 오래된 Python 버전에서 무언가가 깨진다면, 자유롭게 신고하거나 수정 사항을 보내 주세요.</li>
<li>현재 지원되는 STL은 <code>libc++</code>뿐입니다. 공식적으로 지원되는 컴파일러는 <code>clang</code>뿐이지만, 외부 커뮤니티 구성원들이 일반적으로 <code>gcc</code>로도 빌드가 유지되도록 하고 있습니다. 자세한 내용은 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/toolchain_support.md">지원되는 툴체인 문서</a>를 보세요.</li>
</ul>
<p>대부분의 개발은 Ubuntu에서 이루어집니다(Chromium의 빌드 인프라는 현재 22.04, Jammy Jellyfish에서 실행됩니다). 아래에 다른 배포판에 대한 일부 지침이 있지만, 대부분 지원되지 않습니다. 다만 설치 지침은 <a href="#docker">Docker</a>에서 찾을 수 있습니다.</p>
<h2 id="depot_tools-설치"><code>depot_tools</code> 설치</h2><p><code>depot_tools</code> 저장소를 클론하세요.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">git <span class="hljs-built_in">clone</span> https://chromium.googlesource.com/chromium/tools/depot_tools.git</span></code></pre><p><code>depot_tools</code>를 <code>PATH</code>의 맨 앞에 추가하세요(아마 이것을 <code>~/.bashrc</code> 또는 <code>~/.zshrc</code>에 넣고 싶을 것입니다). <code>depot_tools</code>를 <code>/path/to/depot_tools</code>에 클론했다고 가정하면 다음과 같습니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash"><span class="hljs-built_in">export</span> PATH=<span class="hljs-string">"/path/to/depot_tools:<span class="hljs-variable">$PATH</span>"</span></span></code></pre><p><code>depot_tools</code>를 홈 디렉터리에 클론할 때는 <code>PATH</code>에서 <strong><code>~</code>를 사용하지 마세요</strong>. 그렇지 않으면 <code>gclient runhooks</code> 실행에 실패합니다. 대신 <code>$HOME</code> 또는 절대 경로를 사용해야 합니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash"><span class="hljs-built_in">export</span> PATH=<span class="hljs-string">"<span class="hljs-variable">${HOME}</span>/depot_tools:<span class="hljs-variable">$PATH</span>"</span></span></code></pre><h2 id="코드-가져오기">코드 가져오기</h2><p>체크아웃을 위한 <code>chromium</code> 디렉터리를 만들고 그 디렉터리로 이동하세요(전체 경로에 공백이 없기만 하면, 이름은 원하는 대로 붙이고 원하는 위치에 둘 수 있습니다).</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash"><span class="hljs-built_in">mkdir</span> ~/chromium &amp;&amp; <span class="hljs-built_in">cd</span> ~/chromium</span></code></pre><p>depot_tools의 <code>fetch</code> 도구를 실행해 코드와 그 의존성을 체크아웃하세요.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">fetch --nohooks chromium</span></code></pre><p>*** note<br /><strong>NixOS 사용자:</strong> <code>fetch</code> 같은 도구는 Nix 셸 없이는 작동하지 않습니다. <a href="https://chromium.googlesource.com/chromium/src/tools">도구 저장소</a>를 <code>git</code>으로 클론한 다음 <code>nix-shell tools/nix/shell.nix</code>를 실행하세요.</p>
<hr />
<p>전체 저장소 이력이 필요 없다면 <code>fetch</code>에 <code>--no-history</code> 플래그를 추가해 많은 시간을 절약할 수 있습니다.</p>
<p><code>fetch</code>에 <code>--git-cache</code>를 전달하면 훨씬 더 빠르게 만들 수 있습니다. 이 옵션은 처음부터 클론하는 대신 공유된 사전 빌드 스냅샷에서 체크아웃을 시드합니다(그리고 <code>--no-history</code>와 달리 전체 이력을 유지합니다).</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">fetch --git-cache chromium</span></code></pre><p>캐시 디렉터리는 자동으로 선택됩니다(<code>$GIT_CACHE_PATH</code>로 재정의 가능). 이 디렉터리는 가져오는 모든 저장소를 미러링하며(Chromium의 경우 약 30GB), 머신의 모든 체크아웃이 공유합니다. 작업 트리는 객체를 복사하는 대신 이를 참조하므로, 체크아웃별 <code>.git</code>은 작게 유지됩니다.</p>
<p>명령은 빠른 연결에서도 30분 정도 걸릴 수 있으며, 느린 연결에서는 여러 시간이 걸릴 수 있습니다.</p>
<p>머신에 이미 빌드 의존성을 설치했다면(예: 다른 체크아웃에서), <code>--nohooks</code> 플래그를 생략할 수 있으며 <code>fetch</code>가 마지막에 자동으로 <code>gclient runhooks</code>를 실행합니다.</p>
<p><code>fetch</code>가 완료되면 작업 디렉터리에 숨김 <code>.gclient</code> 파일과 <code>src</code>라는 디렉터리가 생성되어 있을 것입니다. 나머지 지침은 <code>src</code> 디렉터리로 이동했다고 가정합니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash"><span class="hljs-built_in">cd</span> src</span></code></pre><h3 id="추가-빌드-의존성-설치">추가 빌드 의존성 설치</h3><p>코드를 체크아웃했고 Ubuntu를 사용한다고 가정하면, <a href="https://chromium.googlesource.com/chromium/src/+/main/build/install-build-deps.sh">build/install-build-deps.sh</a>를 실행하세요.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">./build/install-build-deps.sh</span></code></pre><p>다른 배포판에서는 빌드 의존성을 조정해야 할 수도 있습니다. 이 문서 끝에 몇 가지 <a href="#%EB%8B%A4%EB%A5%B8-%EB%B0%B0%ED%8F%AC%ED%8C%90%EC%97%90-%EB%8C%80%ED%95%9C-%EC%B0%B8%EA%B3%A0-%EC%82%AC%ED%95%AD">참고 사항</a>이 있지만, 그 정확성은 보장하지 않습니다.</p>
<h3 id="훅-실행">훅 실행</h3><p><code>install-build-deps</code>를 적어도 한 번 실행한 뒤에는 Chromium 전용 훅을 실행할 수 있습니다. 이 훅은 필요한 추가 바이너리와 기타 항목을 다운로드합니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">gclient runhooks</span></code></pre><p><em>선택 사항</em>: 빌드가 일부 Google 서비스와 통신하도록 하고 싶다면 <a href="https://www.chromium.org/developers/how-tos/api-keys">API 키 설치</a>도 할 수 있습니다. 하지만 대부분의 개발 및 테스트 목적에는 필요하지 않습니다.</p>

<li><code>Default</code>를 다른 이름으로 바꿀 수 있지만, <code>out</code>의 하위 디렉터리여야 합니다.</li>

</ul>
<h3 id="더-빠른-빌드">더 빠른 빌드</h3><p>이 섹션에는 빌드 속도를 높이기 위해 변경할 수 있는 몇 가지 항목이 들어 있으며, 가장 큰 차이를 만드는 항목이 먼저 오도록 정렬되어 있습니다.</p>
<h4 id="원격-실행-사용">원격 실행 사용</h4><p>*** note<br /><strong>경고:</strong> Google 직원이라면 아래 지침을 따르지 마세요. 대신 <a href="https://goto.google.com/chrome-linux-build#set-up-remote-execution">go/chrome-linux-build#set-up-remote-execution</a>을 보세요.</p>
<hr />
<p>Chromium 빌드는 <a href="https://github.com/bazelbuild/remote-apis">REAPI</a>와 호환되는 원격 실행 시스템을 사용하여 상당히 빨라질 수 있습니다. 이를 통해 원격 캐싱의 이점을 얻고 공유 워커 클러스터에서 많은 빌드 액션을 병렬로 실행할 수 있습니다. Chromium 빌드는 빌드 액션을 원격으로 실행하기 위해 Google이 개발한 <a href="https://pkg.go.dev/go.chromium.org/build/siso#section-readme">Siso</a>라는 클라이언트를 사용합니다.</p>
<p>시작하려면 REAPI 호환 백엔드에 접근할 수 있어야 합니다.</p>
<h5 id="google-rbe">Google RBE</h5><p>다음 지침은 Google로부터 Chromium의 Google RBE 서비스를 사용하라는 초대를 받았고 그 서비스에 대한 접근 권한을 부여받았다고 가정합니다. 그렇지 않다면 대신 <a href="#non-google-rbe">non Google RBE</a>를 보세요.</p>
<p><a href="https://www.chromium.org/getting-involved/become-a-committer/#try-job-access">tryjob 접근 권한</a>이 있는 기여자는 Google이 비용을 지불하는 RBE 백엔드에 접근할 수 있도록 Googler에게 대신 <a href="mailto:accounts@chromium.org">accounts@chromium.org</a>로 이메일을 보내 달라고 요청하세요. 외부 기여자를 위한 원격 실행은 최선 노력(best-effort) 방식이라는 점에 유의하세요. 언제 초대될지는 보장하지 않습니다.</p>
<p>Google의 RBE와 함께 <code>siso</code>를 사용하려면 먼저 다음을 해야 합니다.</p>
<ol>
<li><code>siso login</code>을 실행하고 권한이 부여된 계정으로 로그인합니다. OAuth2 흐름에서 차단된다면 <code>gcloud auth login</code>을 실행하세요(siso v1.3.12부터는 환경 변수 <code>SISO_CREDENTIAL_HELPER=gcloud</code>도 export).</li>
</ol>
<p>다음으로, Chromium 기여자에게 맞는 올바른 인스턴스를 사용하도록 <code>.gclient</code> 구성에 <code>rbe_instance</code>를 지정해야 합니다.</p>
<p>*** note<br /><strong>경고:</strong> Google 직원이라면 아래 지침을 따르지 마세요. 대신 <a href="https://goto.google.com/chrome-linux-build#set-up-remote-execution">go/chrome-linux-build#set-up-remote-execution</a>을 보세요.</p>
<hr />
<pre><code class="hljs language-plaintext">solutions = [
  {
    ...,
    "custom_vars": {
      # This is the correct instance name for using Chromium's RBE service.
      # You can only use it if you were granted access to it. If you use your
      # own REAPI-compatible backend, you will need to set reapi_address,
      # reapi_instance and reapi_backend_config_path instead.
      # see [non Google RBE](#non-google-rbe) below.
      "rbe_instance": "projects/rbe-chromium-untrusted/instances/default_instance",
    },
  },
]</code></pre><p>그리고 <code>gclient sync</code>를 실행하세요. 그러면 방금 <code>.gclient</code> 파일에 추가한 Google RBE를 사용하도록 <code>build/config/siso/.sisoenv</code>와 <code>build/config/siso/backend_config/backend.star</code>의 구성 파일이 다시 생성됩니다.</p>
<h5 id="non-google-rbe">non Google RBE</h5><p>Google의 RBE 백엔드에 접근할 수 없는 다른 사용자는 <a href="https://github.com/bazelbuild/remote-apis#servers">다른 호환 백엔드</a> 중 어떤 것이든 사용할 수 있습니다. 이 경우 인증 방법, 인스턴스 이름 등에 관한 다음 지침을 자신의 백엔드에 맞게 조정해야 합니다.</p>
<ul>
<li>폐쇄망이고 인증이 없다면 환경 변수 <code>RBE_service_no_security=true</code>를 export하세요.</li>
<li>mTLS를 사용한다면 환경 변수 <code>RBE_tls_client_auth_key</code>와 <code>RBE_tls_client_auth_cert</code>를 export하세요.</li>
<li>Google OAuth2를 사용한다면 <code>gcloud</code>를 사용할 수 있습니다. <code>gcloud auth login</code>을 실행하세요(siso v1.3.12부터는 환경 변수 <code>SISO_CREDENTIAL_HELPER=gcloud</code>도 export).</li>
<li>그렇지 않으면 자체 <a href="https://github.com/EngFlow/credential-helper-spec/blob/main/spec.md">credential helper</a>를 사용해야 할 수도 있습니다. 환경 변수 <code>SISO_CREDENTIAL_HELPER=/path/to/your/credhelper</code>를 export하세요.</li>
</ul>
<p>Google RBE가 아닌 자체 REAPI 백엔드의 경우 자체 backend.star를 준비해야 합니다.</p>
<p>예: 큰 워커 풀이 없는 백엔드의 경우</p>
<pre><code class="hljs language-plaintext">load("@builtin//struct.star", "module")

<p>위의 backend.star에 대한 경로 이름인 <code>reapi_address</code>, <code>reapi_instance</code>, <code>reapi_backend_config_path</code>를 설정하세요.</p>
<pre><code class="hljs language-plaintext">solutions = [

<p>팁: reapi 접근을 위한 연결 풀을 제한하기 위해 <code>--reapi_grpc_conn_pool=1</code> 같은 값을 사용하고 싶을 수 있습니다.</p>
<p>팁: 백엔드가 grpc 스트림이 취소되거나 닫힐 때 액션을 종료하고 WaitExecution과 함께 작동하지 않는다면 <code>--reapi_keep_exec_stream</code>을 사용하고 싶을 수 있습니다.</p>
<p>팁: 다음과 같이 이 플래그들을 <code>build/config/siso/.sisorc</code>에 넣을 수 있습니다.</p>
<pre><code class="hljs language-shell">ninja --reapi_grpc_conn_pool=1 --reapi_keep_exec_stream</code></pre><p>그러면 siso가 <code>siso ninja</code> 또는 <code>autoninja</code>에 이 플래그들을 사용합니다.</p>
<h5 id="원격-실행을-위한-gn-설정">원격 실행을 위한 gn 설정</h5><p>그런 다음 <code>args.gn</code>에 다음 GN 인수를 추가하세요.</p>
<pre><code class="hljs language-plaintext">use_remoteexec = true
use_siso = true</code></pre><p><code>args.gn</code>에 <code>use_reclient=true</code>가 들어 있다면 제거하거나 <code>use_reclient=false</code>로 바꾸세요.</p>
<p>이것으로 끝입니다. 아래에 설명된 것처럼 Chromium을 빌드할 때는 항상 <code>siso</code>나 <code>ninja</code>를 직접 호출하지 말고 <code>autoninja</code>를 사용해야 한다는 점을 기억하세요.</p>
<p>원격 실행 사용에 대해 질문이 있으면 <a href="https://groups.google.com/a/chromium.org/g/build">build@chromium.org</a>에 문의하세요.</p>
<h4 id="디버그-심볼을-더-적게-포함하기">디버그 심볼을 더 적게 포함하기</h4><p>기본적으로 GN은 모든 디버그 assertion이 활성화된(<code>is_debug=true</code>) 빌드를 만들고 전체 디버그 정보(<code>symbol_level=2</code>)를 포함합니다. <code>symbol_level=1</code>로 설정하면 스택 트레이스에는 충분하지만 줄 단위 디버깅에는 부족한 정보를 생성합니다. <code>symbol_level=0</code>으로 설정하면 디버그 심볼을 전혀 포함하지 않습니다. 어느 쪽이든 전체 심볼보다 빌드가 빨라집니다.</p>
<h4 id="blink와-v8의-디버그-심볼-비활성화">Blink와 v8의 디버그 심볼 비활성화</h4><p>템플릿을 광범위하게 사용하기 때문에 Blink 코드는 우리 디버그 심볼의 약 절반을 생성합니다. Blink를 디버그할 필요가 전혀 없다면 GN 인수 <code>blink_symbol_level=0</code>을 설정할 수 있습니다. 마찬가지로 v8을 디버그할 필요가 없다면 GN 인수 <code>v8_symbol_level=0</code>을 설정하여 빌드 속도를 개선할 수 있습니다.</p>
<h4 id="icecc-사용">Icecc 사용</h4><p><a href="https://github.com/icecc/icecream">Icecc</a>는 중앙 스케줄러를 통해 빌드 부하를 공유하는 분산 컴파일러입니다. 현재 많은 외부 기여자가 이를 사용합니다. 예: Intel, Opera, Samsung(Siso를 사용하고 있다면 유용하지 않습니다).</p>
<p><code>icecc</code>를 사용하려면 다음 GN 인수를 설정하세요.</p>
<pre><code class="hljs language-plaintext">use_debug_fission=false
is_clang=false</code></pre><p><a href="https://github.com/icecc/icecream/commit/b2ce5b9cc4bd1900f55c3684214e409fa81e7a92">bundled_binutils 제한</a>, <a href="http://gcc.gnu.org/wiki/DebugFission">debug fission 제한</a>에 대한 자세한 내용은 이 링크들을 보세요.</p>
<p>glibc 2.21 이상을 사용할 때는 시스템 링커를 사용하는 것도 필요할 수 있습니다. <a href="https://bugs.debian.org/cgi-bin/bugreport.cgi?bug=808181">관련 버그</a>를 보세요.</p>
<h4 id="ccache">ccache</h4><p><a href="https://ccache.dev">ccache</a>를 사용하여 로컬 빌드 속도를 높일 수 있습니다(역시 Siso를 사용 중이라면 유용하지 않습니다).</p>
<p>작업 디렉터리들이 공통으로 갖는 상위 디렉터리(예: <code>/home/yourusername/development</code>)로 <code>CCACHE_BASEDIR</code>를 설정하여 ccache 히트율을 높이세요. <code>CCACHE_SLOPPINESS=include_file_mtime</code> 사용을 고려하세요(여러 작업 디렉터리를 사용하는 경우, 트리의 svn sync된 부분에서 헤더 시간이 서로 다를 것이기 때문입니다. 추가 정보는 <a href="https://ccache.dev/manual/latest.html#_troubleshooting">ccache 문제 해결 섹션</a>을 보세요). 홈 디렉터리에서 로컬 물리 디스크 디렉터리로 향하는 심볼릭 링크를 사용해 작업 개발 디렉터리에 접근한다면, 다음을 <code>.bashrc</code>에 넣는 것을 고려하세요.</p>
<pre><code class="hljs language-plaintext">alias cd="cd -P"</code></pre><p>그러면 <code>$PWD</code> 또는 <code>cwd</code>가 항상 논리 디렉터리가 아니라 물리 디렉터리를 가리키게 됩니다(그리고 <code>CCACHE_BASEDIR</code>도 물리 상위 디렉터리를 가리키는지 확인하세요).</p>
<p>ccache를 올바르게 튜닝하면, trunk를 추적하는 브랜치를 사용하고 trunk에 최신 상태이며 거의 같은 시점에 gclient sync된 두 번째 작업 디렉터리는 약 1/3 시간 안에 chrome을 빌드할 수 있어야 하며, <code>ccache -s</code>가 보고하는 캐시 미스는 거의 증가하지 않아야 합니다.</p>
<p>이는 <a href="http://git-scm.com/docs/git-worktree">git-worktree</a>를 사용하고 여러 로컬 작업 디렉터리를 동시에 유지하는 경우 특히 유용합니다.</p>
<h4 id="tmpfs-사용">tmpfs 사용</h4><p>빌드 출력에 tmpfs를 사용하여 필요한 디스크 쓰기 양을 줄일 수 있습니다. 즉, 빌드 출력이 들어가는 출력 디렉터리에 tmpfs를 마운트합니다.</p>
<p>root로 실행:</p>
<pre><code class="hljs language-plaintext">mount -t tmpfs -o size=20G,nr_inodes=40k,mode=1777 tmpfs /path/to/out</code></pre><p>*** note<br /><strong>주의:</strong> tmpfs를 뒷받침할 만큼 충분한 RAM + 스왑이 필요합니다. 전체 디버그 빌드의 경우 약 20GB가 필요합니다. chrome 타깃만 빌드하거나 릴리스 빌드의 경우 더 적게 필요합니다.</p>
<hr />
<p>HP Z600(Intel core i7, 하이퍼스레딩된 16코어, 12GB RAM)에서의 빠르고 대략적인 벤치마크 수치</p>
<ul>
<li>tmpfs 사용:<ul>
<li>12m:20s</li>
</ul>
</li>
<li>tmpfs 미사용<ul>
<li>15m:40s</li>
</ul>
</li>
</ul>
<h3 id="더-작은-빌드">더 작은 빌드</h3><p>Chrome 바이너리는 기본적으로 내장 심볼을 포함합니다. Linux <code>strip</code> 명령을 사용해 이 디버그 정보를 제거하면 크기를 줄일 수 있습니다. 또한 GN 인수 <code>is_official_build = true</code>로 공식 빌드 모드를 활성화하면 바이너리 크기를 줄이고 모든 최적화를 켤 수 있습니다.</p>
<h2 id="chromium-빌드">Chromium 빌드</h2><p>다음 명령으로 Siso 또는 Ninja를 사용해 Chromium("chrome" 타깃)을 빌드하세요.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">autoninja -C out/Default chrome</span></code></pre><p>(<code>autoninja</code>는 <code>siso</code> 또는 <code>ninja</code>에 전달되는 인수에 대해 최적 값을 자동으로 제공하는 래퍼입니다.)</p>
<p>명령줄에서 <code>gn ls out/Default</code>를 실행하면 GN의 다른 모든 빌드 타깃 목록을 얻을 수 있습니다. 하나를 컴파일하려면 앞의 "//" 없이 GN 레이블을 Siso/Ninja에 전달하세요(예: <code>//chrome/test:unit_tests</code>의 경우 <code>autoninja -C out/Default chrome/test:unit_tests</code> 사용).</p>
<p>팁: <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/siso_tips.md">Siso 팁</a>을 보세요.</p>
<h2 id="chromium-실행">Chromium 실행</h2><p>빌드가 끝나면 브라우저를 간단히 실행할 수 있습니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">out/Default/chrome</span></code></pre><p>Chrome Remote Desktop을 지원하는 원격 머신을 사용 중이라면 이것을 .bashrc / .bash_profile에 추가할 수 있습니다.</p>
<pre><code class="hljs language-shell">if [[ -z "${DISPLAY}" ]]; then
<span class="hljs-meta prompt_">  # </span><span class="language-bash">In reality, Chrome Remote Desktop starts with 20 and increases <span class="hljs-keyword">until</span> it</span>
<span class="hljs-meta prompt_">  # </span><span class="language-bash">finds an available ID [1]. So this isn<span class="hljs-string">'t guaranteed to always work, but</span></span>
<span class="hljs-meta prompt_">  # </span><span class="language-bash"><span class="hljs-string">should work on the vast majoriy of cases.</span></span>
<span class="hljs-meta prompt_">  #</span><span class="language-bash"><span class="hljs-string">
  # [1] https://source.chromium.org/chromium/chromium/src/+/main:remoting/host/linux/linux_me2me_host.py;l=112;drc=464a632e21bcec76c743930d4db8556613e21fd8</span></span>
  export DISPLAY=:20
fi</code></pre><p>이는 SSH 세션에서 Chrome을 실행하면 UI 출력이 Chrome Remote Desktop에서 사용 가능해진다는 뜻입니다.</p>
<h2 id="테스트-타깃-실행">테스트 타깃 실행</h2><p>테스트는 타입과 디렉터리 구조에서 존재하는 위치에 따라 여러 테스트 타깃으로 나뉩니다. 특정 단위 테스트 또는 브라우저 테스트 파일이 어떤 타깃에 해당하는지 보려면 다음 명령을 사용할 수 있습니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">gn refs out/Default --testonly=<span class="hljs-literal">true</span> --<span class="hljs-built_in">type</span>=executable --all chrome/browser/ui/browser_unittest.cc</span>
//chrome/test:unit_tests</code></pre><p>위 예에서 타깃은 unit_tests입니다. unit_tests 바이너리는 다음 명령을 실행해 빌드할 수 있습니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">autoninja -C out/Default unit_tests</span></code></pre><p>unit_tests 바이너리를 실행하여 테스트를 실행할 수 있습니다. 또한 <code>--gtest_filter</code> 인수를 사용해 어떤 테스트를 실행할지 제한할 수도 있습니다. 예:</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">out/Default/unit_tests --gtest_filter=<span class="hljs-string">"BrowserListUnitTest.*"</span></span></code></pre><p>GoogleTest에 대한 자세한 내용은 <a href="https://github.com/google/googletest">GitHub 페이지</a>에서 확인할 수 있습니다.</p>
<h2 id="체크아웃-업데이트">체크아웃 업데이트</h2><p>기존 체크아웃을 업데이트하려면 다음을 실행할 수 있습니다.</p>
<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash">git rebase-update</span>
<span class="hljs-meta prompt_">$ </span><span class="language-bash">gclient <span class="hljs-built_in">sync</span></span></code></pre><p>첫 번째 명령은 기본 Chromium 소스 저장소를 업데이트하고 로컬 브랜치를 tip-of-tree(일명 Git 브랜치 <code>origin/main</code>) 위로 rebase합니다. 이 스크립트를 사용하고 싶지 않다면 <code>git pull</code> 또는 다른 일반적인 Git 명령을 사용해 저장소를 업데이트해도 됩니다.</p>
<p>두 번째 명령은 의존성을 적절한 버전으로 동기화하고 필요에 따라 훅을 다시 실행합니다.</p>
<h2 id="팁-트릭-문제-해결">팁, 트릭, 문제 해결</h2><h3 id="링커-크래시">링커 크래시</h3><p>최종 링크 단계에서 다음과 같은 상황일 때:</p>
<pre><code class="hljs language-plaintext">LINK out/Debug/chrome</code></pre><p>다음과 같은 오류가 발생하면:</p>
<pre><code class="hljs language-plaintext">collect2: ld terminated with signal 6 Aborted terminate called after throwing an instance of 'std::bad_alloc'
collect2: ld terminated with signal 11 [Segmentation fault], core dumped</code></pre><p>또는:</p>
<pre><code class="hljs language-plaintext">LLVM ERROR: out of memory</code></pre><p>링크 중 메모리가 부족할 가능성이 높습니다. 빌드하려면 64비트 시스템을 <em>반드시</em> 사용해야 합니다. 다음 빌드 설정을 시도해 보세요(다른 설정은 <a href="https://www.chromium.org/developers/gn-build-configuration">GN 빌드 구성</a>을 보세요).</p>
<ul>
<li>릴리스 모드로 빌드(디버깅 심볼은 더 많은 메모리를 요구합니다):<br />  <code>is_debug = false</code></li>
<li>심볼 끄기: <code>symbol_level = 0</code></li>
<li>컴포넌트 모드로 빌드(개발 전용이며, 더 느리고 기능이 깨질 수 있습니다): <code>is_component_build = true</code></li>
<li>Linux에서 공식(ThinLTO) 빌드를 하는 경우 vm.max_map_count 커널 파라미터를 늘리세요. <code>vm.max_map_count</code> 값을 기본값(예: 65530)에서 예를 들어 262144로 늘리세요. 현재 세션에서 셸로 <code>sudo sysctl -w vm.max_map_count=262144</code> 명령을 실행해 설정할 수 있고, 영구 저장하려면 <code>/etc/sysctl.conf</code>에 <code>vm.max_map_count=262144</code>를 추가할 수 있습니다.</li>
</ul>

<h2 id="다른-배포판에-대한-참고-사항">다른 배포판에 대한 참고 사항</h2><h3 id="arch-linux">Arch Linux</h3><p>빌드 의존성을 설치하기 위해 <code>install-build-deps.sh</code>를 실행하는 대신 다음을 실행하세요.</p>

<pre><code class="hljs language-shell"><span class="hljs-meta prompt_">$ </span><span class="language-bash"><span class="hljs-built_in">sudo</span> install-build-deps.sh --no-arm</span></code></pre><h3 id="fedora">Fedora</h3><p><code>build/install-build-deps.sh</code>를 실행하는 대신 다음을 실행하세요.</p>

<h3 id="gentoo">Gentoo</h3><p>그냥 <code>emerge www-client/chromium</code>을 실행하면 됩니다.</p>
<h3 id="nixos">NixOS</h3><p>개발 환경이 있는 셸을 얻으려면:</p>
<pre><code class="hljs language-sh">$ nix-shell tools/nix/shell.nix</code></pre><p>개발 환경에서 명령을 실행하려면:</p>
<pre><code class="hljs language-sh">$ NIX_SHELL_RUN=<span class="hljs-string">'autoninja -C out/Default chrome'</span> nix-shell tools/nix/shell.nix</code></pre><p>원격 인덱싱 지원이 있는 clangd를 설정하려면 아래 명령을 실행한 다음 경로를 편집기 설정에 복사하세요.</p>
<pre><code class="hljs language-sh">$ NIX_SHELL_RUN=<span class="hljs-string">'readlink /usr/bin/clangd'</span> nix-shell tools/nix/shell.nix</code></pre><h3 id="opensuse">OpenSUSE</h3><p>의존성을 설치하려면 <code>zypper</code> 명령을 사용하세요.</p>
<p>(openSUSE 11.1 이상)</p>
<pre><code class="hljs language-shell">sudo zypper in subversion pkg-config python perl bison flex gperf \
     mozilla-nss-devel glib2-devel gtk-devel wdiff lighttpd gcc gcc-c++ \
     mozilla-nspr mozilla-nspr-devel php5-fastcgi alsa-devel libexpat-devel \
     libjpeg-devel libbz2-devel</code></pre><p>11.0의 경우 <code>mozilla-nspr</code>와 <code>mozilla-nspr-devel</code> 대신 <code>libnspr4-0d</code>와 <code>libnspr4-dev</code>를 사용하고, <code>php5-fastcgi</code> 대신 <code>php5-cgi</code>를 사용하세요.</p>
<p>(openSUSE 11.0)</p>
<pre><code class="hljs language-shell">sudo zypper in subversion pkg-config python perl \
     bison flex gperf mozilla-nss-devel glib2-devel gtk-devel \
     libnspr4-0d libnspr4-dev wdiff lighttpd gcc gcc-c++ libexpat-devel \
     php5-cgi alsa-devel gtk3-devel jpeg-devel</code></pre><p>Ubuntu 패키지 <code>sun-java6-fonts</code>에는 사용되는 글꼴 중 Java 글꼴의 일부가 들어 있습니다. 이 패키지는 어차피 Java를 전제 조건으로 요구하므로, 동등한 openSUSE Sun Java 패키지를 설치해 같은 일을 할 수 있습니다.</p>
<pre><code class="hljs language-shell">sudo zypper in java-1_6_0-sun</code></pre><p>WebKit은 현재 Microsoft 글꼴에 하드 링크되어 있습니다. <code>zypper</code>를 사용해 이를 설치하려면:</p>
<pre><code class="hljs language-shell">sudo zypper in fetchmsttfonts pullin-msttf-fonts</code></pre><p>위에서 설치한 글꼴은 Ubuntu용으로 경로가 하드코딩되어 있으므로, 작동하게 하려면 적절한 위치로 심볼릭 링크를 만드세요.</p>
<pre><code class="hljs language-shell">sudo mkdir -p /usr/share/fonts/truetype/msttcorefonts

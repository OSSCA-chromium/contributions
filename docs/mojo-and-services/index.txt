<p>이 문서는 <strong>Intro to Mojo and Services</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/mojo_and_services.md"><code>docs/mojo_and_services.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<h2 id="개요">개요</h2><p>이 문서는 개발자가 Chromium에서 Mojo를 효과적으로 사용하기 시작하는 데 필요한 최소한의 정보를 담고 있으며, 예제 Mojo 인터페이스 사용법, 서비스 정의와 연결, 그리고 Content 계층의 핵심 서비스에 대한 간단한 개요를 포함한다.</p>
<p>입문 가이드, API 레퍼런스 등은 다른 <a href="https://chromium.googlesource.com/chromium/src/+/main/docs/README.md#Mojo-Services">Mojo 및 Services</a> 문서를 참고하라.</p>
<h2 id="mojo-용어">Mojo 용어</h2><p><strong>메시지 파이프(message pipe)</strong> 는 한 쌍의 <strong>엔드포인트(endpoint)</strong> 이다. 각 엔드포인트에는 수신 메시지 큐가 있으며, 한 엔드포인트에 메시지를 쓰면 사실상 그 메시지가 다른 쪽(<strong>피어(peer)</strong>) 엔드포인트에 큐잉된다. 따라서 메시지 파이프는 양방향이다.</p>
<p><strong>mojom</strong> 파일은 <strong>인터페이스(interface)</strong> 를 기술하며, 인터페이스는 강한 타입을 갖는 <strong>메시지(message)</strong> 컬렉션이다. Google protobuf에 익숙한 개발자에게는, 각 인터페이스 메시지가 대략 하나의 proto 메시지와 유사하다.</p>
<p>mojom 인터페이스와 메시지 파이프가 주어지면, 엔드포인트 중 하나를 <strong><code>Remote</code></strong> 로 지정할 수 있으며, 이는 인터페이스가 기술하는 메시지를 <em>보내는</em> 데 사용된다. 다른 엔드포인트는 <strong><code>Receiver</code></strong> 로 지정할 수 있으며, 인터페이스 메시지를 <em>받는</em> 데 사용된다.</p>
<p>*** aside<br />참고: 위 일반화는 약간 지나치게 단순화한 것이다. 메시지 파이프는 여전히 양방향이며, mojom 메시지가 응답을 기대할 수도 있다는 점을 기억하라. 응답은 <code>Receiver</code> 엔드포인트에서 전송되고 <code>Remote</code> 엔드포인트에서 수신된다.</p>
<hr />
<p>수신된 메시지를 처리하려면 <code>Receiver</code> 엔드포인트가 해당 mojom 인터페이스의 <strong>구현(implementation)</strong> 과 연결되어 있어야 한다. (<em>즉,</em> <strong>바인드(bound)</strong> 되어 있어야 한다.) 수신된 메시지는 구현 객체의 대응되는 인터페이스 메서드를 호출하는 예약된 태스크로 디스패치된다.</p>
<p>이 모든 것을 생각하는 또 다른 방식은 단순히 다음과 같다. <strong><code>Remote</code>는 대응되는 원격 <code>Receiver</code>와 연결된 자기 인터페이스의 원격 구현에 대해 호출을 수행한다.</strong></p>
<h2 id="예제-새-frame-인터페이스-정의하기">예제: 새 Frame 인터페이스 정의하기</h2><p>이를 Chrome에 적용해 보자. 렌더 프레임에서 브라우저 프로세스의 대응되는 <code>RenderFrameHostImpl</code> 인스턴스로 "Ping" 메시지를 보내고 싶다고 가정하자. 이를 위해 적절한 mojom 인터페이스를 정의하고, 그 인터페이스를 사용할 파이프를 만들고, 전송된 메시지가 그곳에서 수신 및 처리될 수 있도록 파이프의 한쪽 끝을 올바른 위치까지 연결해야 한다. 이 절에서는 그 과정을 자세히 살펴본다.</p>
<h3 id="인터페이스-정의하기">인터페이스 정의하기</h3><p>첫 번째 단계는 다음과 같이 인터페이스 정의가 들어 있는 새 <code>.mojom</code> 파일을 만드는 것이다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// src/example/public/mojom/pingable.mojom</span>
<span class="hljs-keyword">module</span> example.mojom;

interface Pingable {
  <span class="hljs-comment">// "Ping"을 수신하고 임의의 정수로 응답한다.</span>
  <span class="hljs-built_in">Ping</span>() =&gt; (int32 random);
};</code></pre><p>여기에는 이 정의에 대한 C++ 바인딩을 생성하기 위한 대응되는 빌드 규칙이 있어야 한다.</p>
<pre><code class="hljs language-python"><span class="hljs-comment"># src/example/public/mojom/BUILD.gn</span>
<span class="hljs-keyword">import</span>(<span class="hljs-string">"//mojo/public/tools/bindings/mojom.gni"</span>)
mojom(<span class="hljs-string">"mojom"</span>) {
  sources = [ <span class="hljs-string">"pingable.mojom"</span> ]
}</code></pre><h3 id="파이프-만들기">파이프 만들기</h3><p>이제 이 인터페이스를 사용할 메시지 파이프를 만들어 보자.</p>
<p>*** aside<br />일반적인 규칙이자 Mojo 사용 시 편의를 위한 관례로, 인터페이스의 <em>클라이언트</em> (<em>즉,</em> <code>Remote</code> 쪽)가 보통 새 파이프를 만드는 주체다. 이는 <code>Remote</code>가 InterfaceRequest 엔드포인트가 어디론가 전달되거나 바인드되기를 기다리지 않고도 즉시 메시지 전송을 시작하는 데 사용될 수 있기 때문에 편리하다.</p>
<hr />
<p>이 코드는 렌더러 어딘가에 배치될 것이다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// src/third_party/blink/example/public/pingable.h</span>
mojo::Remote&lt;example::mojom::Pingable&gt; pingable;
mojo::PendingReceiver&lt;example::mojom::Pingable&gt; receiver =
    pingable.<span class="hljs-built_in">BindNewPipeAndPassReceiver</span>();</code></pre><p>이 예제에서 <code>pingable</code>은 <code>Remote</code>이고, <code>receiver</code>는 <code>PendingReceiver</code>다. <code>PendingReceiver</code>는 나중에 결국 <code>Receiver</code>로 바뀔 <code>Receiver</code>의 전 단계 객체다. <code>BindNewPipeAndPassReceiver</code>는 메시지 파이프를 만드는 가장 일반적인 방식이다. 이 함수는 반환값으로 <code>PendingReceiver</code>를 산출한다.</p>
<p>*** aside<br />참고: <code>PendingReceiver</code>는 실제로 아무것도 <strong>하지</strong> 않는다. 이는 단일 메시지 파이프 엔드포인트를 담는 비활성 홀더다. 이것은 오직 그 엔드포인트가 같은 인터페이스 타입의 <code>Receiver</code>에 의해 바인드될 것으로 기대된다는 점을 나타내어, 컴파일 시점에 엔드포인트의 타입을 더 강하게 만들기 위해 존재한다.</p>
<hr />
<h3 id="메시지-보내기">메시지 보내기</h3><p>마지막으로, <code>Remote</code>에서 <code>Ping()</code> 메서드를 호출하여 메시지를 보낼 수 있다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// src/third_party/blink/example/public/pingable.h</span>
pingable-&gt;<span class="hljs-built_in">Ping</span>(base::<span class="hljs-built_in">BindOnce</span>(&amp;OnPong));</code></pre><p>*** aside<br /><strong>중요:</strong> 응답을 받고 싶다면 <code>OnPong</code>이 호출될 때까지 <code>pingable</code> 객체를 살아 있게 유지해야 한다. 결국 <code>pingable</code>은 자기 메시지 파이프 엔드포인트를 <em>소유</em>한다. 이것이 파괴되면 엔드포인트도 파괴되며, 응답 메시지를 받을 대상이 아무것도 남지 않는다.</p>
<hr />
<p>거의 끝났다! 물론 모든 것이 이렇게 쉬웠다면 이 문서가 존재할 필요도 없었을 것이다. 우리는 렌더러 프로세스에서 브라우저 프로세스로 메시지를 보내는 어려운 문제를, 위의 <code>receiver</code> 객체를 가져와 브라우저 프로세스에 어떻게든 전달하여 그곳에서 수신 메시지를 디스패치하는 <code>Receiver</code>로 바꿀 수 있게 하는 문제로 변환했다.</p>
<h3 id="브라우저에-pendingreceiver-보내기">브라우저에 <code>PendingReceiver</code> 보내기</h3><p><code>PendingReceiver</code>들(그리고 일반적으로 메시지 파이프 엔드포인트들)은 mojom 메시지를 통해 자유롭게 전송할 수 있는 또 다른 종류의 객체일 뿐이라는 점에 주목할 가치가 있다. <code>PendingReceiver</code>를 어딘가로 가져가는 가장 일반적인 방법은 이미 연결된 다른 인터페이스의 메서드 인자로 전달하는 것이다.</p>
<p>렌더러의 <code>RenderFrameImpl</code>과 브라우저의 대응되는 <code>RenderFrameHostImpl</code> 사이에 항상 연결되어 있는 이러한 인터페이스 중 하나가 <a href="https://cs.chromium.org/chromium/src/third_party/blink/public/mojom/browser_interface_broker.mojom"><code>BrowserInterfaceBroker</code></a>다. 이 인터페이스는 다른 인터페이스를 획득하기 위한 팩토리다. 이 인터페이스의 <code>GetInterface</code> 메서드는 임의의 인터페이스 receiver를 전달할 수 있게 해 주는 <code>GenericPendingReceiver</code>를 받는다.</p>
<pre><code class="hljs language-cpp">interface BrowserInterfaceBroker {
  <span class="hljs-built_in">GetInterface</span>(mojo_base.mojom.GenericPendingReceiver receiver);
}</code></pre><p><code>GenericPendingReceiver</code>는 어떤 특정 <code>PendingReceiver</code>로부터도 암시적으로 생성될 수 있으므로, 앞서 <code>BindNewPipeAndPassReceiver</code>를 통해 만든 <code>receiver</code> 객체로 이 메서드를 호출할 수 있다.</p>
<pre><code class="hljs language-cpp">RenderFrame* my_frame = <span class="hljs-built_in">GetMyFrame</span>();

<h2 id="서비스-개요-및-용어">서비스 개요 및 용어</h2><p>이전 절은 Chromium에서 Mojo IPC가 어떻게 사용되는지의 겉면만 살짝 다룬 것이다. 렌더러-브라우저 메시징은 단순하고 순수한 코드 양만 놓고 보면 아마도 가장 널리 쓰이는 사용 방식이지만, 우리는 전통적인 Content 브라우저/렌더러/gpu/유틸리티 프로세스 분할보다 조금 더 세분화된 서비스 집합으로 코드베이스를 점진적으로 분해하고 있다.</p>
<p><strong>서비스(service)</strong> 는 하나 이상의 관련 기능 또는 동작을 구현하는 자체 완결적인 코드 라이브러리이며, 외부 코드와의 상호작용은 일반적으로 브라우저 프로세스가 중개하는 Mojo 인터페이스 연결을 통해서만 <em>독점적으로</em> 이루어진다.</p>
<p>각 서비스는 브라우저가 서비스 인스턴스를 관리하는 데 사용할 수 있는 주 Mojo 인터페이스를 정의하고 구현한다.</p>
<h2 id="예제-단순한-out-of-process-서비스-만들기">예제: 단순한 Out-of-Process 서비스 만들기</h2><p>Chromium에서 새 서비스를 준비하고 실행하기 위해 일반적으로 필요한 단계는 여러 가지가 있다.</p>
<ul>
<li>주 서비스 인터페이스와 구현을 정의한다</li>
<li>out-of-process 코드에서 구현을 연결한다</li>
<li>서비스 프로세스를 시작하기 위한 브라우저 로직을 작성한다</li>
</ul>
<p>이 절에서는 간단한 설명과 함께 이 단계들을 차례로 살펴본다. 여기에서 사용되는 개념과 API에 대한 더 자세한 문서는 <a href="https://chromium.googlesource.com/chromium/src/+/main/mojo/README.md">Mojo</a> 문서를 참고하라.</p>

<h3 id="서비스-시작하기">서비스 시작하기</h3><p>서비스를 in-process로 실행한다면, 실제로 더 할 만한 흥미로운 일은 없다. 다른 객체와 마찬가지로 서비스 구현을 인스턴스화할 수 있지만, 동시에 out-of-process인 것처럼 Mojo Remote를 통해 이 구현과 대화할 수도 있다.</p>
<p>이전 절에서 수행한 연결 이후 out-of-process 서비스 인스턴스를 시작하려면, Content의 <a href="https://cs.chromium.org/chromium/src/content/public/browser/service_process_host.h?rcl=e7a1f6c9a24f3151c875598174a05167fb12c5d5&amp;l=47"><code>ServiceProcessHost</code></a> API를 사용하라.</p>

<h3 id="샌드박스-지정하기">샌드박스 지정하기</h3><p>모든 서비스는 샌드박스를 지정해야 한다. 이상적으로 서비스는 운영체제 리소스에 접근할 필요가 없는 한 <code>kService</code> 프로세스 샌드박스 안에서 실행된다. 커스텀 샌드박스가 필요한 서비스의 경우 <a href="mailto:security-dev@chromium.org">security-dev@chromium.org</a>와 상의하여 새 샌드박스 타입을 정의해야 한다.</p>
<p>인터페이스의 샌드박스를 정의하는 선호 방식은 <code>.mojom</code> 파일 안의 <code>interface {}</code>에 <code>[ServiceSandbox=type]</code> 속성을 지정하는 것이다.</p>
<pre><code class="hljs language-plaintext">import "sandbox/policy/mojom/sandbox.mojom";
[ServiceSandbox=sandbox.mojom.Sandbox.kService]
interface FakeService {
  ...
};</code></pre><p>유효한 값은 <a href="https://cs.chromium.org/chromium/src/sandbox/policy/mojom/sandbox.mojom"><code>//sandbox/policy/mojom/sandbox.mojom</code></a>에 있는 값들이다. 샌드박스는 인터페이스가 <code>content::ServiceProcessHost::Launch()</code>를 사용해 out-of-process로 시작되는 경우에만 적용된다는 점에 유의하라.</p>
<p>최후의 수단으로, 기반 플랫폼 샌드박스에 대한 동적 또는 기능 기반 매핑을 달성할 수는 있지만, 이를 위해서는 ContentBrowserClient를 통한 배관 작업이 필요하다. (예: <code>ShouldSandboxNetworkService()</code>)</p>
<h2 id="content-계층-서비스-개요">Content 계층 서비스 개요</h2><h3 id="인터페이스-브로커">인터페이스 브로커</h3><p>우리는 렌더러의 frame 객체와 브라우저 프로세스의 대응되는 <code>RenderFrameHostImpl</code> 사이에 지속적인 연결을 갖는 명시적인 mojom 인터페이스를 정의한다. 이 인터페이스는 <a href="https://cs.chromium.org/chromium/src/third_party/blink/public/mojom/browser_interface_broker.mojom?rcl=09aa5ae71649974cae8ad4f889d7cd093637ccdb&amp;l=11"><code>BrowserInterfaceBroker</code></a>라고 하며, 작업하기가 꽤 쉽다. <code>RenderFrameHostImpl</code>에 새 메서드를 추가하면 된다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">RenderFrameHostImpl::GetGoatTeleporter</span><span class="hljs-params">(
    mojo::PendingReceiver&lt;magic::mojom::GoatTeleporter&gt; receiver)</span> </span>{
  goat_teleporter_receiver_.<span class="hljs-built_in">Bind</span>(std::<span class="hljs-built_in">move</span>(receiver));
}</code></pre><p>그리고 이 메서드를 <code>browser_interface_binders.cc</code>의 <code>PopulateFrameBinders</code> 함수에 등록한다. 이 함수는 특정 인터페이스들을 각각의 host에 있는 핸들러로 매핑한다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// //content/browser/browser_interface_binders.cc</span>
<span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">PopulateFrameBinders</span><span class="hljs-params">(RenderFrameHostImpl* host,
                          mojo::BinderMap* map)</span> </span>{
...
  map-&gt;<span class="hljs-built_in">Add</span>&lt;magic::mojom::GoatTeleporter&gt;(base::<span class="hljs-built_in">BindRepeating</span>(
      &amp;RenderFrameHostImpl::GetGoatTeleporter, base::<span class="hljs-built_in">Unretained</span>(host)));
}</code></pre><p>태스크 러너를 지정하여 다른 sequence에서 인터페이스를 바인드하는 것도 가능하다.</p>
<pre><code class="hljs language-cpp"><span class="hljs-comment">// //content/browser/browser_interface_binders.cc</span>
<span class="hljs-function"><span class="hljs-type">void</span> <span class="hljs-title">PopulateFrameBinders</span><span class="hljs-params">(RenderFrameHostImpl* host,
                          mojo::BinderMap* map)</span> </span>{
...
  map-&gt;<span class="hljs-built_in">Add</span>&lt;magic::mojom::GoatTeleporter&gt;(base::<span class="hljs-built_in">BindRepeating</span>(
      &amp;RenderFrameHostImpl::GetGoatTeleporter, base::<span class="hljs-built_in">Unretained</span>(host)),
      <span class="hljs-built_in">GetIOThreadTaskRunner</span>({}));
}</code></pre><p>Worker들도 렌더러와 브라우저 프로세스의 대응되는 원격 구현 사이에 <code>BrowserInterfaceBroker</code> 연결을 가진다. 새 worker 전용 인터페이스를 추가하는 것은 위에서 frame에 대해 자세히 설명한 단계와 유사하지만, 다음과 같은 차이가 있다.</p>
<ul>
<li>Dedicated Worker의 경우, <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/worker_host/dedicated_worker_host.h"><code>DedicatedWorkerHost</code></a>에 새 메서드를 추가하고 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/browser_interface_binders.cc;l=1126;drc=e24e0a914ff0da18e78044ebad7518afe9e13847"><code>PopulateDedicatedWorkerBinders</code></a>에 등록한다</li>
<li>Shared Worker의 경우, <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/worker_host/shared_worker_host.h"><code>SharedWorkerHost</code></a>에 새 메서드를 추가하고 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/browser_interface_binders.cc;l=1232;drc=e24e0a914ff0da18e78044ebad7518afe9e13847"><code>PopulateSharedWorkerBinders</code></a>에 등록한다</li>
<li>Service Worker의 경우, <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/service_worker/service_worker_host.h"><code>ServiceWorkerHost</code></a>에 새 메서드를 추가하고 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/browser_interface_binders.cc;l=1326;drc=e24e0a914ff0da18e78044ebad7518afe9e13847"><code>PopulateServiceWorkerBinders</code></a>에 등록한다</li>
</ul>
<p>인터페이스는 렌더러의 Blink <code>Platform</code> 객체와 브라우저 프로세스의 대응되는 <code>RenderProcessHost</code> 객체 사이의 <code>BrowserInterfaceBroker</code> 연결을 사용하여 프로세스 수준에서도 추가될 수 있다. 이렇게 하면 렌더러 안의 어떤 스레드(frame 및 worker 스레드 포함)든 해당 인터페이스에 접근할 수 있지만, 사용되는 <code>BrowserInterfaceBroker</code> 구현이 스레드 안전해야 하므로 추가 오버헤드가 따른다. 새 프로세스 수준 인터페이스를 추가하려면, <code>RenderProcessHostImpl</code>에 새 메서드를 추가하고 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/renderer_host/render_process_host_impl.cc;l=2317;drc=a817d852ea2f2085624d64154ad847dfa3faaeb6"><code>RenderProcessHostImpl::RegisterMojoInterfaces</code></a> 안에서 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/renderer_host/render_process_host_impl.h;l=918;drc=ec5eaba0e021b757d5cbbf2c27ac8f06809d81e9"><code>AddUIThreadInterface</code></a>를 호출하여 등록한다. 렌더러 쪽에서는 <a href="https://source.chromium.org/chromium/chromium/src/+/main:third_party/blink/public/platform/platform.h;l=781;drc=ee1482552c4c97b40f15605fe6a52565cfe74548"><code>Platform::GetBrowserInterfaceBroker</code></a>를 사용하여 대응되는 <code>BrowserInterfaceBroker</code> 객체를 얻고, 그 객체에서 <code>GetInterface</code>를 호출한다.</p>
<p>embedder 전용 document-scoped 인터페이스를 바인드하려면, <a href="https://cs.chromium.org/chromium/src/content/public/browser/content_browser_client.h?rcl=3eb14ce219e383daa0cd8d743f475f9d9ce8c81a&amp;l=999"><code>ContentBrowserClient::RegisterBrowserInterfaceBindersForFrame()</code></a>을 오버라이드하고 제공된 map에 binder들을 추가한다.</p>
<p>*** aside<br />참고: BrowserInterfaceBroker가 요청된 인터페이스에 대한 binder를 찾을 수 없으면, 관련 context host에서 <code>ReportNoBinderForInterface()</code>를 호출하며, 이는 host의 receiver에서 <code>ReportBadMessage()</code> 호출로 이어진다. (그 결과 중 하나는 렌더러 종료다.) 테스트에서 이러한 크래시를 피하려면(content_shell이 일부 Chrome 전용 인터페이스를 바인드하지 않지만 렌더러가 어쨌든 그것들을 요청하는 경우), <code>browser_interface_binders.cc</code>의 <a href="https://cs.chromium.org/chromium/src/content/browser/browser_interface_binders.cc?rcl=12e73e76a6898cb6df6a361a98320a8936f37949&amp;l=407"><code>EmptyBinderForFrame</code></a> 헬퍼를 사용하라. 그러나 가능하다면 렌더러 쪽과 브라우저 쪽을 일관되게 유지하는 것이 권장된다.</p>

<h3 id="navigation-associated-인터페이스">Navigation-Associated 인터페이스</h3><p>서로 다른 frame에서 오는 메시지들의 순서가 중요한 경우, 그리고 메시지들이 navigation을 구현하는 메시지들과 관련하여 올바르게 정렬되어야 하는 경우에는 navigation-associated 인터페이스를 사용할 수 있다. Navigation-associated 인터페이스는 각 frame에서 대응되는 <code>RenderFrameHostImpl</code> 객체로 가는 연결을 활용하며, navigation과 관련된 메시지에 사용되는 동일한 FIFO 파이프를 통해 각 연결의 메시지를 보낸다. 그 결과 navigation 이후에 전송된 메시지는 navigation 관련 메시지 이후에 브라우저 프로세스에 도착하는 것이 보장되며, 같은 document의 서로 다른 frame에서 전송된 메시지들의 순서도 보존된다.</p>
<p>새 navigation-associated 인터페이스를 추가하려면, <code>RenderFrameHostImpl</code>에 새 메서드를 만들고 <a href="https://source.chromium.org/chromium/chromium/src/+/main:content/browser/renderer_host/render_frame_host_impl.cc;l=8365;drc=a817d852ea2f2085624d64154ad847dfa3faaeb6"><code>RenderFrameHostImpl::SetUpMojoConnection</code></a>에서 <code>associated_registry_-&gt;AddInterface</code> 호출로 이를 등록한다. 렌더러에서는 <a href="https://source.chromium.org/chromium/chromium/src/+/main:third_party/blink/renderer/core/frame/local_frame.h;l=409;drc=19f17a30e102f811bc90a13f79e8ad39193a6403"><code>LocalFrame::GetRemoteNavigationAssociatedInterfaces</code></a>를 사용하여 <code>GetInterface</code>를 호출할 객체를 얻는다. (이 호출은 pending receiver 대신 <a href="https://chromium.googlesource.com/chromium/src/+/main/mojo/public/cpp/bindings/README.md#associated-interfaces">pending associated receiver</a>를 받는다는 점을 제외하면 <code>BrowserInterfaceBroker::GetInterface</code>와 유사하다.)</p>

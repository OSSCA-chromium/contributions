<p>이 문서는 <strong>Integrating a feature with the Origin Trials framework</strong>(<a href="https://chromium.googlesource.com/chromium/src/+/main/docs/origin_trials_integration.md"><code>docs/origin_trials_integration.md</code></a>) 문서의 한국어 전체 번역입니다.</p>
</blockquote>
<p><a href="https://googlechrome.github.io/OriginTrials/developer-guide.html">Origin Trials framework</a>를 통해 기능을 노출하려면 몇 가지 코드 변경이 필요하다.</p>
<p>*** note<br /><strong>경고:</strong> 이는 Blink에서 구현된 기능에만 사용할 수 있다.</p>
<hr />
<h2 id="코드-변경">코드 변경</h2><p>*** promo<br /><strong>참고:</strong> origin trial 실행을 요청하기 전에 이러한 코드 변경을 랜딩할 수 있다.<br />이 코드 변경은 origin trial을 통해 기능을 제어할 수 있게 하지만, origin trial 승인이 필요하지는 않다. 프로세스에 대한 자세한 내용은 <a href="https://www.chromium.org/blink/origin-trials/running-an-origin-trial">Running an Origin Trial</a>을 참조하라.</p>
<hr />
<h3 id="1단계-origin-trial을-위해-blink에-runtime-enabled-feature-추가">1단계: Origin Trial을 위해 Blink에 Runtime Enabled Feature 추가</h3><p>먼저 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/runtime_enabled_features.json5"><code>runtime_enabled_features.json5</code></a>를 구성해야 한다. 아직 Blink의 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/RuntimeEnabledFeatures.md">Runtime Enabled Feature</a> 플래그가 없다면 이 파일에 항목을 추가해야 한다.</p>
<p>항목의 다음 필드들이 관련 있다.</p>
<ul>
<li><code>name</code>: runtime enabled feature의 이름. 예: <code>"MyFeature"</code>.</li>
<li><code>origin_trial_feature_name</code>: origin trial에서 사용할 runtime enabled feature의 이름. runtime feature 플래그, 즉 <code>name</code> 필드와 같을 수도 있고 다를 수도 있다. 결국 이 구성된 이름은 origin trials 개발자 콘솔에서 사용된다.</li>
<li><code>origin_trial_os</code>: trial 활성화를 허용할 플랫폼의 <code>[list]</code>를 지정한다. 목록 값은 대소문자를 구분하지 않지만, 정의된 <code>OS_&lt;platform&gt;</code> 매크로 중 하나와 일치해야 한다(<a href="https://chromium.googlesource.com/chromium/src/+/main/build/build_config.h"><code>build_config.h</code></a> 참조).</li>
<li><code>origin_trial_allows_third_party</code>: 서드파티 토큰이 올바르게 동작하도록 허용하려면 활성화해야 한다. 서드파티 매칭을 지원하려는 경우에만 true로 설정한다.</li>
<li><code>base_feature</code>: 값이 <code>"none"</code>이 아니면 <code>blink::features</code> 네임스페이스에 <code>base::Feature</code>를 생성한다. Origin Trial을 원격으로 제어하는 데 도움이 된다. <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/RuntimeEnabledFeatures.md#generate-a-instance-from-a-blink-feature">Blink Feature에서 <code>base::Feature</code> 인스턴스 생성</a>도 참조하라.</li>
</ul>
<p>Origin Trial에만 한정되지 않는 항목:</p>
<ul>
<li><code>status</code>: runtime enabled feature가 Blink에서 언제 활성화되는지 제어한다. <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/RuntimeEnabledFeatures.md#adding-a-runtime-enabled-feature">the Status table</a>도 참조하라.</li>
<li><code>base_feature_status</code>: <code>base_feature</code>가 정의한 <code>base::Feature</code>가 언제 활성화되는지 제어한다.</li>
</ul>
<p>자세한 내용은 json5 파일 및 위에 링크된 문서에 설명되어 있다.</p>
<p>runtime enabled feature 플래그가 <a href="#1-c%EC%97%90%EC%84%9C">C++에서 사용되는 경우</a>, <code>RuntimeEnabledFeatures::MyFeatureEnabled()</code>의 인수 없는 오버로드를 호출하는 모든 호출자를 <code>const FeatureContext*</code>를 받는 오버로드로 변경해야 한다. 여기에 <code>ExecutionContext</code>를 전달할 수 있다. 예를 들어 <code>ExecutionContext::From(ScriptState*)</code>를 사용한다.</p>
<h4 id="예시">예시</h4><p>RuntimeEnabledFeature 플래그 이름, trial 이름, <code>base::Feature</code>가 모두 같은 경우:</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  name<span class="hljs-punctuation">:</span> <span class="hljs-string">"MyFeature"</span><span class="hljs-punctuation">,</span>  <span class="hljs-comment">// Generates `RuntimeEnabledFeatures::MyFeatureEnabled()`</span>

  <span class="hljs-comment">// No need to specify base_feature.</span>
<span class="hljs-punctuation">}</span><span class="hljs-punctuation">,</span></code></pre><p>RuntimeEnabledFeature 플래그 이름, trial 이름, <code>base::Feature</code> 이름이 서로 다른 경우:</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  name<span class="hljs-punctuation">:</span> <span class="hljs-string">"MyFeature"</span><span class="hljs-punctuation">,</span>

<span class="hljs-punctuation">}</span><span class="hljs-punctuation">,</span></code></pre><p>특정 플랫폼으로 제한된 trial:</p>
<pre><code class="hljs language-json"><span class="hljs-punctuation">{</span>
  name<span class="hljs-punctuation">:</span> <span class="hljs-string">"MyFeature"</span><span class="hljs-punctuation">,</span>
  origin_trial_feature_name<span class="hljs-punctuation">:</span> <span class="hljs-string">"MyFeature"</span><span class="hljs-punctuation">,</span>
  origin_trial_os<span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">"android"</span><span class="hljs-punctuation">]</span><span class="hljs-punctuation">,</span>
  status<span class="hljs-punctuation">:</span> <span class="hljs-string">"experimental"</span><span class="hljs-punctuation">,</span>
<span class="hljs-punctuation">}</span><span class="hljs-punctuation">,</span></code></pre><h4 id="webview-고려-사항">WebView 고려 사항</h4><p>WebView는 <code>"android"</code> OS 타깃의 일부로 빌드되므로, Android에서 활성화된 trial을 WebView에서 제외하는 것은 불가능하다.</p>
<p>trial 대상 기능을 다른 Android 플랫폼과 함께 WebView에서도 활성화할 수 있다면, 이 방식이 선호된다.</p>
<p>그것이 가능하지 않은 상황에서는 <a href="https://chromium.googlesource.com/chromium/src/+/main/android_webview/lib/aw_main_delegate.cc"><code>aw_main_delegate.cc</code></a>의 <code>AwMainDelegate::BasicStartupComplete()</code>에서 비활성화할 trial 이름을 값으로 하여 <code>embedder_support::kOriginTrialDisabledFeatures</code> 스위치를 추가함으로써 origin trial을 명시적으로 비활성화하는 것이 권장되는 해결책이다.</p>
<p>이를 수행하는 방법의 예시는 <a href="https://crrev.com/c/3733267">https://crrev.com/c/3733267</a> 를 참조하라.</p>
<h3 id="2단계-접근-게이트-처리">2단계: 접근 게이트 처리</h3><p>구성이 완료되면 origin trial 뒤에서 기능 접근을 게이트하는 메커니즘이 두 가지 있다. 기능 구현에 적절하게 둘 중 하나 또는 둘 모두를 사용할 수 있다.</p>
<h4 id="1-c에서">1) C++에서</h4><p>런타임에 기능을 노출하기 위해 Blink 코드에서 호출할 수 있는 네이티브 C++ 메서드:</p>
<pre><code class="hljs language-cpp"><span class="hljs-function"><span class="hljs-type">bool</span> <span class="hljs-title">RuntimeEnabledFeatures::MyFeatureEnabled</span><span class="hljs-params">(ExecutionContext*)</span></span></code></pre><p>*** note<br /><strong>경고:</strong> 기능 구현은 enabled 검사 결과를 영속화해서는 안 된다. 코드는 기능 접근을 게이트하기 위해 필요한 만큼 자주 단순히 <code>RuntimeEnabledFeatures::MyFeatureEnabled(ExecutionContext*)</code>를 호출해야 한다.</p>
<hr />
<h4 id="2-1-web-idl에서">2-1) Web IDL에서</h4><p>JavaScript 메서드/속성/객체를 노출하고 숨기는 코드를 자동 생성하는 데 사용할 수 있는 IDL 속성 [<a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/bindings/IDLExtendedAttributes.md#RuntimeEnabled">RuntimeEnabled</a>].</p>
<pre><code class="hljs language-cpp">[RuntimeEnabled=MyFeature]
partial interface Navigator {
     readonly attribute MyFeatureManager myFeature;
}</code></pre><h4 id="2-2-css-속성">2-2) CSS 속성</h4><p>*** promo<br /><strong>참고:</strong> CSS 속성의 경우 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/core/css/css_style_declaration.idl">CSSStyleDeclaration</a>에서의 노출이 런타임에 처리되므로 IDL 파일을 편집할 필요가 없다.</p>
<hr />
<p>origin trial로 새 CSS 속성에 대한 실험도 실행할 수 있다. 위와 같이 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/runtime_enabled_features.json5"><code>runtime_enabled_features.json5</code></a>에서 기능을 구성한 다음 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/core/css/css_properties.json5"><code>css_properties.json5</code></a>로 이동한다. 파일에 설명된 대로, 방금 정의한 기능과 CSS 속성을 연결하기 위해 <code>runtime_flag</code>를 사용한다. 그러면 CSS 속성이 runtime feature에 정의된 origin trial에 자동으로 연결된다. trial이 활성화되면 JavaScript(<code>Element.style</code>)와 CSS(<code>@supports</code> 포함) 양쪽에서 사용할 수 있게 된다.</p>
<p>*** promo<br /><strong>예시:</strong> <a href="https://chromium.googlesource.com/chromium/src/+/ff2ab8b89745602c8300322c2a0158e210178c7e/third_party/blink/renderer/core/css/css_properties.json5#2635">origin-trial-test-property</a>는 runtime feature <code>OriginTrialsSampleAPI</code>를 통해 제어되고 이어서 <code>Frobulate</code>라는 origin trial로 제어되는 테스트 CSS 속성을 정의한다.</p>
<hr />
<p>*** note<br /><strong>이슈:</strong> 드문 경우지만 CSS style declaration 이후에 origin trial 토큰이 스크립트를 통해 추가되면 CSS 속성은 활성화되어 완전히 동작하지만 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/core/css/css_style_declaration.idl">CSSStyleDeclaration</a> 인터페이스에는 나타나지 않는다. 즉, <code>Element.style</code>에서 접근할 수 없다. 이 이슈는 crbug/1041993에서 추적된다.</p>

<h3 id="3단계-runtime-enabled-feature를-basefeature에-매핑하기선택-사항">3단계: Runtime Enabled Feature를 <code>base::Feature</code>에 매핑하기(선택 사항)</h3><p>다음 예시가 있다고 하자.</p>

</blockquote>
<p>*** note<br /><strong>요약:</strong> <code>MyFeature</code>를 켜도 <code>blink::features::kMyFeature</code>가 자동으로 켜지지 않으며, 그 반대도 마찬가지다.</p>

<h4 id="옵션-1-basefeature를-완전히-활성화하기-예-kmyfeature">옵션 1: <code>base::Feature</code>를 완전히 활성화하기. 예: <code>kMyFeature</code></h4><p>그리고 위 인용문에서 제안한 것처럼 Origin Trial이 기능(runtime enabled feature 플래그 <code>blink::features::MyFeature</code>를 통해)을 언제 사용할 수 있는지 결정하게 한다. <code>base::Feature</code>는 원격 Finch 구성으로 활성화하거나 C++의 기본값을 업데이트하여 활성화할 수 있다.</p>
<p>하지만 Origin Trial이 종료된 뒤에는, <code>MyFeature</code>가 제어하는 부분을 독립적으로 활성화할 수 없다면 Finch로 기능을 점진적으로 확대하는 것이 불가능해진다. 예를 들어 새로운 Web API <code>MyFeatureAPI</code>가 있는 경우, <code>MyFeature</code>를 활성화하면 Blink/브라우저 구현 없이 IDL만 모든 사람에게 제공될 수 있다.</p>

<h3 id="5단계-web-test-추가">5단계: Web Test 추가</h3><p>[<a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/bindings/IDLExtendedAttributes.md#RuntimeEnabled">RuntimeEnabled</a>] IDL 속성을 사용할 때는 V8 bindings 코드가 예상대로 동작하는지 확인하기 위해 web test를 추가해야 한다. 기능이 어떻게 노출되는지에 따라 노출된 인터페이스에 대한 테스트뿐 아니라 스크립트로 추가된 토큰에 대한 테스트도 필요하다. 예시는 <a href="https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/web_tests/http/tests/origin_trials/webexposed/">origin_trials/webexposed</a>의 기존 테스트를 참조하라.</p>
<h2 id="제한-사항">제한 사항</h2><p>이 origin trial의 특성 때문에 할 수 없는 것은, 현재 페이지/컨텍스트에서 기능이 사용될지 여부를 브라우저 또는 렌더러 시작 시점에 아는 것이다. 이는 시작하기 위해 많은 비용이 드는 처리가 필요한 경우(예를 들어 사용자의 하드 드라이브를 인덱싱하거나, 흥미로운 날씨 패턴을 찾기 위해 도시 전체를 스캔하는 경우) 혹시 사용될 수 있으므로 <em>모든</em> 사용자에 대해 브라우저 시작 시 수행하거나, 최초 접근 시 수행해야 함을 의미한다. (최초 접근 방식을 선택하면 실험을 시도하는 사람들만 지연을 느끼게 되며, 바라건대 처음 사용할 때만 느끼게 된다.) 우리는 <code>OriginTrials::myFeatureShouldInitialize()</code> 같은 메서드를 제공하는 방안을 조사하고 있다. 이 메서드는 시작 초기화를 해야 하는지 힌트를 줄 것이다. 예를 들어 사용량 때문에 철회되었거나 스로틀된 trial, 전체 origin trials 프레임워크가 비활성화되었는지 등에 대한 검사를 포함할 수 있다. 이 메서드는 보수적으로 동작하여 초기화가 필요하다고 가정하겠지만, 일부 알려진 시나리오에서는 비용이 큰 시작 작업을 피할 수 있다.</p>
<p>마찬가지로 브라우저 프로세스에서 기능을 활성화해야 하는지 알아야 한다면, 렌더러가 런타임에 이를 알려주게 하거나, 아니면 항상 활성화되어 있다고 가정하고 렌더러에서 기능 접근을 게이트해야 한다.</p>
<h2 id="수동-테스트">수동 테스트</h2><p>개발 중 origin trial 기능을 테스트하려면 다음 단계를 따른다.</p>
<ol>
<li><p><a href="https://chromium.googlesource.com/chromium/src/+/main/tools/origin_trials/generate_token.py"><code>generate_token.py</code></a>를 사용하여 테스트 private key로 서명된 토큰을 생성한다. 테스트를 돕기 위해 필요한 어떤 origin에 대해서도 서명된 토큰을 생성할 수 있으며, localhost 또는 127.0.0.1도 포함된다. 예:</p>
<pre><code class="hljs language-bash">tools/origin_trials/generate_token.py http://localhost:8000 MyFeature</code></pre><p>서드파티 토큰을 생성하고, 만료 날짜를 설정하고, 다른 옵션을 제어하기 위한 추가 플래그가 있다. 자세한 내용은 명령 도움말(<code>--help</code>)을 참조하라. 예를 들어 <a href="https://docs.google.com/document/d/1xALH9W7rWmX0FpjudhDeS2TNTEOXuPn4Tlc9VmuPdHA/edit#heading=h.myaz1twlipw">user subset exclusion</a>과 함께 서드파티 토큰을 생성하려면:</p>
<pre><code class="hljs language-bash">tools/origin_trials/generate_token.py --is-third-party --usage-restriction=subset http://localhost:8000 MyFeature</code></pre></li>
<li><p>출력 끝에서 토큰을 복사하여 <a href="https://github.com/jpchase/OriginTrials/blob/gh-pages/developer-guide.md">Developer Guide</a>에 설명된 대로 <code>&lt;meta&gt;</code> 태그 또는 <code>Origin-Trial</code> 헤더에서 사용한다.</p>
</li>

<p><code>content_shell</code>에서는 <code>--origin-trial-public-key</code> 스위치가 필요하지 않다. 기본적으로 테스트 public key를 사용하기 때문이다.</p>
<p>테스트 private key는 repo의 <code>tools/origin_trials/eftest.key</code>에 저장되어 있다. Origin Trials unit test와 web test에서도 사용된다.</p>
<p>명령줄 스위치를 설정할 수 없는 경우(예: Chrome OS)에는 <a href="https://chromium.googlesource.com/chromium/src/+/main/chrome/common/origin_trials/chrome_origin_trial_policy.cc"><code>chrome_origin_trial_policy.cc</code></a>를 직접 수정할 수도 있다.</p>
<p>origin trial 토큰 파싱에 대한 추가 정보(실패 이유 또는 성공한 토큰의 토큰 이름 포함)를 보려면 다음 스위치를 추가할 수 있다.</p>
<p>  <code>--vmodule=trial_token=2,origin_trial_context=1</code></p>
<p><code>is_debug=false</code>로 빌드하는 경우에는 빌드 옵션에 <code>dcheck_always_on=true</code>도 추가해야 하며, 명령줄에 다음을 추가해야 한다.</p>
<p>  <code>--enable-logging=stderr</code></p>
